Modem 2x — validation des boucles de tracking 2026-05-15 — branche feat/modem-2x — méthodologie séquentielle injection → estimation → divergence, stop-at-first-bug.

Le sondeur de canal a été validé (cf sounder_echo_audit.html) et le simulateur nbfm_channel_sim.py reproduit le σ² post-demod OTA (cf modem_2x_reference.html). On utilise maintenant le sim déterministe comme banc de test pour comparer, boucle de tracking par boucle de tracking, la valeur injectée à la valeur estimée par le RX 2x. Phase B — Boucle DRIFT PPM : injecté vs estimé

Instrumentation

Trois modifications minimes pour exposer le drift estimé :

  • rust/modem-core2x/src/rx_v4.rs : ajouté champ pub final_drift_ppm: Option<f64> à RxResult2x.
  • rust/modem-core2x/src/rx2x_session.rs : emit_finalised() recopie cached_drift_ppm (le drift estimé du domaine audio RAW, avant correction par cubic resampling).
  • rust/modem-cli/src/main.rs : nouvelle ligne stderr Drift estimate: drift_ppm=X (from N validated SOFs).
Piège évité : instrumenter en faisant estimate_drift_from_sof_positions(&result.validated_sof_positions, …) renvoie ~0 par construction. validated_sof_positions contient les positions SOF dans le flux de symboles POST-correction (post cubic-resample). Le drift résiduel après correction est ~0. La vraie estimation est cached_drift_ppm (calculée sur l'audio RAW).

Côté Python : study/snr_sweep_2x.py parse cette ligne (regex _RX_LINE_RE_DRIFT), ajoute les colonnes injected_drift_ppm + estimated_drift_ppm + validated_sofs à la CSV. Nouveau script study/plot_drift_vs_estimate.py trace l'injecté vs l'estimé pour les 3 profils.

Deux modes d'échec stackés :
  1. Drift ≤ ~15 ppm : non détecté. cached_drift_ppm = 0, pas de cubic resample appliqué. Gardner closed-loop doit absorber seul. À 10 ppm il commence à craquer (HIGH++ casse).
  2. Drift ≥ 20 ppm : détecté correctement (29.6 à 30 ppm — match à 0.5 ppm près). Mais le décodage casse quand même (σ² explose).
Le bug n'est pas dans l'estimation du drift. Il est dans la compensation downstream.

σ² monte progressivement DANS chaque cycle (×30 entre CW 0 et CW 9), reset à chaque PLHEADER. Pattern en dents-de-scie. σ²_T (tangentielle) domine — c'est de la rotation de phase qui accumule, pas du bruit AM-AM.

Calcul : drift D = 30 ppm × fc downmix 1500 Hz = 0.045 Hz d'écart résiduel après cubic resample imparfait. Sur 4 s de cycle, rotation = 2π × 0.045 × 4 = 1.13 rad = 65° au dernier CW du cycle. Voilà le σ²_T qui explose.

Cause racine

Le cubic resample audio corrige le sample timing, mais NE compense PAS la rotation de phase porteuse induite par le drift × fc. Après downmix par exp(-j·2π·fc·t), le signal baseband porte une résidu de rotation ω = 2π·fc·ppm. À 1500 Hz × 30 ppm = 16 °/sec, qui accumule à 65° par cycle 4 s.

Comparaison V3 (qui marche OTA à 30 ppm) : V3 estime le drift global et resample le buffer audio entier avant downmix. La rotation de phase induite est donc absorbée par le resample. 2x fait le contraire — il cherche d'abord à matcher symbole par symbole avec Gardner + cubic resample chunk-par-chunk, ce qui laisse la rotation.

Diagnostic — quantification entière des positions SOF

Première borne du bug : la résolution de l'estimateur LS sur des positions SOF entières.

L'algorithme estimate_drift_from_sof_positions() (rx_v4.rs:302-339) fait un LS slope-fit sur les positions SOF entières dans le flux de symboles. Pour résoudre un drift D ppm, il faut au moins 1 symbole de drift cumulé sur la longueur du burst : 

n_cycles × cycle_period_sym × D × 1e-6 ≥ 1

Pour un burst de 10 ko :

Profilcyclescycle_period_symDrift seuil (1 sym cumul)
HIGH2X7~6000≈ 24 ppm
HIGH+2X6~4500≈ 37 ppm
HIGH++2X5~4300≈ 47 ppm

Sous ce seuil, le drift cumulé est moins d'un symbole entier — l'estimateur retourne ~0 ou n'update pas la cache (seuil de mise à jour = 0.5 ppm). Le sweep confirme : à drift ≤ 10 ppm, cached_drift_ppm reste à 0 sur les 3 profils.

Au-dessus du seuil (drift ≥ 20 ppm), l'estimation tient bien (29.6 ppm pour 30 ppm injecté). Mais le décodage casse quand même à cause du second mode d'échec (rotation de phase non compensée — voir résultats sweep ci-dessus).

Cause racine #1 — quantification entière SOF

Positions SOF entières → résolution LS limitée à 1 / (n_cycles × cycle_period) ≈ 25-50 ppm sur un burst de 10 ko. Sous ce seuil, drift non détecté, pas de cubic resample, le Gardner closed-loop doit absorber seul.

Cause racine #2 — rotation de phase porteuse non compensée

Au-dessus du seuil de détection, le drift est correctement estimé mais le cubic resample audio ne corrige que le sample timing — la rotation de phase porteuse induite (ω = 2π·fc·ppm = 2π·1500·30e-6 ≈ 0.28 rad/s, soit 65° par cycle 4 s à 30 ppm) reste. C'est ce que mesure σ²_T qui explose intra-cycle dans le log RX_V4_LOG_CW=1.

V3 ne souffre pas de ce problème parce que sa stratégie est différente : il resample le buffer audio en bulk avant downmix. La rotation de phase est absorbée par le resample lui-même. 2x fait un cubic resample chunk-par-chunk après estimation, mais le downmix par fc a déjà été appliqué — la phase porteuse résiduelle n'est plus rattrapée par le resample.

Fix Phase 1 — Backward fix au 2ème SOF (L5)

Implémenté 2026-05-15. Stratégie : quand estimate_drift_from_raw met à jour cached_drift_ppm de ≥ 0.5 ppm AFTER que la session a déjà décodé des cycles (= des SOFs ont été validés), invalider l'état de scan FSM et re-décoder depuis le sample 0 du audio_buffer (qui n'est jamais trimmé).

Reset minimal

Premier essai : reset complet (FSM + app_header + cw_bytes + accumulateurs). Échec : LDPC convergeait spuriement post-reset sur la META-CW (σ²=1e-3 floor) mais decode_meta_payload retournait None (bytes garbage). Le full reset détruisait les CWs déjà décodés correctement (notamment la META @ cycle 0 qui décode OK avant que la rotation de phase ne casse les cycles suivants).

Deuxième essai — reset minimal, livré : on garde cw_bytes, app_header, session_id, pls_anchors, on reset seulement state→Idle, scan_cursor_abs=0, validated_sof_positions, cycles, σ² accumulators, FFE taps. Les CWs ré-décodés post-reset se mergent dans cw_bytes (insert/overwrite) à mesure que la nouvelle passe décode correctement.
