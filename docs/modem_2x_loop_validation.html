Modem 2x — validation des boucles de tracking 2026-05-15 — branche feat/modem-2x — méthodologie séquentielle injection → estimation → divergence, stop-at-first-bug.

Le sondeur de canal a été validé (cf sounder_echo_audit.html) et le simulateur nbfm_channel_sim.py reproduit le σ² post-demod OTA (cf modem_2x_reference.html). On utilise maintenant le sim déterministe comme banc de test pour comparer, boucle de tracking par boucle de tracking, la valeur injectée à la valeur estimée par le RX 2x. Approche stop-at-first-divergence.

Phase B — Boucle DRIFT PPM : injecté vs estimé

Instrumentation

Trois modifications minimes pour exposer le drift estimé :

  • rust/modem-core2x/src/rx_v4.rs : ajouté champ pub final_drift_ppm: Option<f64> à RxResult2x.
  • rust/modem-core2x/src/rx2x_session.rs : emit_finalised() appelle estimate_drift_from_sof_positions() et stocke le résultat.
  • rust/modem-cli/src/main.rs : nouvelle ligne stderr Drift estimate: drift_ppm=X (from N validated SOFs).

Côté Python : study/snr_sweep_2x.py parse cette ligne (regex _RX_LINE_RE_DRIFT), ajoute les colonnes injected_drift_ppm + estimated_drift_ppm + validated_sofs à la CSV. Nouveau script study/plot_drift_vs_estimate.py trace l'injecté vs l'estimé pour les 3 profils.

Diagnostic — quantification entière des positions SOF

L'algorithme estimate_drift_from_sof_positions() (rx_v4.rs:302-339) fait un LS slope-fit sur les positions SOF entières dans le flux de symboles. Pour résoudre un drift D ppm, il faut au moins 1 symbole de drift cumulé sur la longueur du burst : 

n_cycles × cycle_period_sym × D × 1e-6 ≥ 1

Pour un burst de 10 ko :

Profilcyclescycle_period_symDrift seuil (1 sym cumul)
HIGH2X7~6000≈ 24 ppm
HIGH+2X6~4500≈ 37 ppm
HIGH++2X5~4300≈ 47 ppm

Sous ce seuil, le drift cumulé est moins d'un symbole entier — l'estimateur ne peut pas le résoudre et retourne 0. C'est exactement ce qu'on observe : à drift = 10 ppm, HIGH+2X et HIGH++2X retournent 0 (sous leur seuil de résolution), tandis que HIGH2X retourne 8.8 (juste au seuil, alignement chanceux).

Cause racine du bug drift PPM : les positions SOF sont entières (samples au taux symbole). L'algorithme LS slope-fit ne peut pas résoudre des drifts inférieurs à 1 / (n_cycles × cycle_period) ≈ 25-50 ppm pour un burst de 10 ko. Au-delà, l'estimation est correcte mais arrive trop tard — le Gardner closed-loop en streaming a déjà accumulé une dérive symbole significative.

Pourquoi 2x casse même quand l'estimé matche

Le drift est appliqué par le sim via cubic resampling au taux audio AVANT le modem. La boucle Gardner closed-loop dans streaming_frontend.rs est censée le suivre symbole par symbole. Avec K_p = 0.05, K_i = 0.005 (slow loop DVB-S2X), elle converge en ~200 symboles. Mais pour un drift de 30 ppm, le shift cumulé sur les 200 premiers symboles est 0.006 samples — Gardner converge mais ne corrige presque rien à ce stade.

Le problème combiné :

  1. Bootstrap drift insuffisamment précis — LS-fit retourne 0 ou hop-skip alors qu'on a 10-30 ppm réels.
  2. Gardner closed-loop trop lent — converge en ~200 sym, mais Q_i = 0.005 ⇒ stationary error ≈ 30e-6 × K_i / K_p² ≈ 6e-3 sample par symbole résiduel, large devant la précision de la décision symbole.
  3. Pas de boucle thermal-tracking explicite — la dérive oscillante n'est pas modélisée séparément du drift statique.

Candidats fix

OptionEffortBénéfice attendu
A. SOF position sub-sample via interpolation parabolique du pic Faible (~50 lignes dans find_next_sof) Résolution drift × 10-20. Drifts ≥ 1 ppm résolvables sur 5 cycles.
B. Gardner gain bumped initial + drift bootstrap from PLHEADER 0 Moyen Convergence Gardner plus rapide ; mais bootstrap depuis PLHEADER 0 nécessite un anchor exact, pas trivial.
C. Drift forward Kalman par cycle (vs LS slope-fit final) Moyen Estimation drift à chaque PLHEADER avec lissage temporel → suit aussi le thermal.
D. Bulk resample audio avec drift estimé par chunk + retry decode Élevé Approche V3 portée sur 2x : robuste mais re-traite l'audio plusieurs fois.

Recommandation : A + C en cascade. A donne la précision instantanée, C donne le tracking temporel.

Prochaines étapes

STOP Phases C-F. Le bug drift PPM explique le breakdown 2x observé en Phase A. Pas la peine d'instrumenter les autres boucles tant que celle-ci n'est pas fixée.

  1. Implémenter fix A (SOF sub-sample interpolation parabolique).
  2. Re-runner le sweep drift après fix → vérifier que l'estimé matche l'injecté en dessous de 50 ppm.
  3. Implémenter fix C (Kalman par cycle) si A seul ne suffit pas.
  4. Quand drift est OK : reprendre le sweep Phase A (2x_breakdown_discovery) — si 2x décode propre, alors les autres boucles n'ont pas besoin d'instrumentation tout de suite. Sinon Phase C (phase porteuse).
