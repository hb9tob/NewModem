1. Pourquoi cette architecture — le bug originel

Avant ce refactor, la chaîne RX SDR (Pluto comme SDRplay) était : I/Q → démodulateur FM → décimation audio. Le discriminateur tournait au taux d'échantillonnage I/Q complet (576 kHz, BW 1.5 MHz côté SDRplay) et n'importe quel canal voisin dans cette bande contribuait à l'audio démodulé. Deux bugs concrets en découlaient :

  • Pas de réjection canaux adjacents — un voisin à 25 kHz, même 40 dB plus fort, passait dans le démod sans atténuation. En bande 2 m amateur active, l'audio reçu est inintelligible.
  • DC spike sur le SDRplay — le tuner Mirics MSi001 en zero-IF a une pointe de DC (fuite LO + offset ADC). Le LO étant programmé sur la fréquence utilisateur, la porteuse à recevoir tombait pile sur la pointe. Empiriquement, l'utilisateur obtenait une bien meilleure réception en saisissant 145.500 MHz au lieu de 145.550 MHz (= LO décalé par erreur, signal sortant du DC spike).

Le refactor introduit la sélection canal avant le démod (port du bloc freq_xlating_fir_filter_ccf de GNU Radio) et un LO offset matériel + NCO digital côté SDRplay pour s'écarter du DC spike. La chaîne est partagée entre Pluto et SDRplay : un seul code DSP, paramétré par le backend.

2. Les couches : transport vs DSP

Invariant architectural : ajouter un backend (RTL-SDR, Lime, …) consiste à écrire un seul rx.rs qui fait le pont format-sample → Complex32 et appelle NbfmRxChain::process(). La DSP n'est plus à recopier.

3. La chaîne NbfmRxChain en détail

Topologie GR équivalente

La chaîne est l'idiome canonique de réception NBFM en GNU Radio : un freq_xlating_fir_filter_ccf en amont de nbfm_rx. Notre NbfmRxChain fusionne les deux dans un seul struct, avec le même découpage de responsabilités : 

# GNU Radio flowgraph idiomatique pour NBFM
src = sdr_source(...)            # I/Q at e.g. 576 kS/s
sel = freq_xlating_fir_filter_ccf(
        decim=12,
        taps=firdes.low_pass(...),  # Kaiser 80 dB
        center_freq=lo_offset_hz,
        sampling_freq=576_000,
)
nbfm = nbfm_rx(audio_rate=48_000, quad_rate=48_000, max_dev=5_000)
self.connect(src, sel, nbfm, audio_sink)

Côté Rust, ces quatre lignes deviennent un seul appel : 

let mut chain = NbfmRxChain::new(NbfmRxChainConfig::new(
    input_rate_hz: 576_000,
    max_deviation_hz: 5_000.0,
    lo_offset_hz: 75_000.0,    // 0.0 sur Pluto (DC compensation matérielle)
));
let audio = chain.process(&iq_complex32);  // → Vec<f32> @ 48 kHz

4. Le bloc FreqXlatingFir (port GR)

C'est la pièce nouvelle, et la plus importante. Elle fait trois opérations en un seul bloc :

  1. NCO digital — multiplie chaque échantillon I/Q par exp(−j·ω·n) où ω = 2π·lo_offset/fs. Ramène un signal qui était à +lo_offset Hz vers le DC.
  2. Filtre LPF complexe — FIR à coefficients réels (taps Kaiser, ~700 taps pour 80 dB) appliqué sur l'I/Q déjà descendu en bande de base. Rejette tout ce qui sort du canal NBFM choisi.
  3. Décimation polyphase ÷N — n'évalue le FIR qu'une fois sur N échantillons d'entrée, où N = input_rate / 48 kHz. Le coût de calcul total est dominé par la sortie audio (48 kHz × N_taps × 2 mults complexes).

Pourquoi un seul bloc et pas trois étapes séparées

On pourrait écrire NCO → LPF → décim comme trois opérations distinctes, mais l'astuce de GR (que FreqXlatingFir reproduit) c'est que la décim N divise le coût du FIR par N : le FIR n'est évalué qu'une fois sur N. Et garder les taps réels (au lieu de pré-rotater par exp(j·ω·k) comme GR le fait en interne) coûte deux multiplications par paire (re, im) au lieu de quatre. Au final, ~5 Mmac/s sur le préféré 576 kS/s ÷12, sous 1 % d'un cœur Pi 5.

5. Filtre canal — paramètres Kaiser

Le LPF dans FreqXlatingFir est conçu par kaiser_sinc_taps() à partir de quatre paramètres :

  • fs_hz — taux d'entrée du FIR (= input_rate du SDR).
  • passband_end_hz — dernier Hz à passer ≤ 0.1 dB ; calculé comme max_deviation_hz + AUDIO_MARGIN_HZ (= 3 kHz).
  • stopband_start_hz — premier Hz à atténuer ≤ −80 dB ; passband_end × TRANSITION_RATIO (= 1.5).
  • stopband_db — atténuation cible (default 80 dB).

La largeur de bande totale du canal (sur Complex32, donc symétrique autour de DC) est 2 × passband_end_hz, ce qui correspond à la formule de Carson : 

BW_canal = 2 · (Δf + f_audio_max)
ModeΔf (max_dev)passband_end BW totale (Carson)stopband N taps @ 576 kHz
NBFM standard±5 kHz8 kHz 16 kHz12 kHz~723
narrow NFM±2.5 kHz5.5 kHz 11 kHz8.25 kHz~1051
Pourquoi Kaiser et pas Hamming ? Hamming plafonne à ~53 dB de réjection quel que soit le nombre de taps — insuffisant pour un voisin 60 dB plus fort en bande encombrée. Kaiser permet de viser n'importe quel rejet (formule : β = 0.1102·(A−8.7) pour A > 50 dB) au prix d'un tap count proportionnel.

6. Spécificités backend — LO offset SDRplay

L'avantage architectural : c'est la même NbfmRxChain dans les deux backends. Quand lo_offset_hz = 0, le NCO multiplie par exp(0) = 1 à chaque échantillon — pas de bypass spécial, le code est partagé bit-pour-bit.

Comment c'est paramétré côté code

// SDRplay : modem-sdrplay/src/device.rs
pub const DEFAULT_LO_OFFSET_HZ: i32 = 75_000;

// Programmation du tuner Mirics
tuner_params.rfFreq.rfHz =
    (config.rf_freq_hz as i64 + DEFAULT_LO_OFFSET_HZ as i64) as f64;

// Construction de la chaîne (modem-sdrplay/src/rx.rs)
let chain = NbfmRxChain::new(NbfmRxChainConfig::new(
    host_iq_rate_hz as u32,
    session.config.max_deviation_hz,
    DEFAULT_LO_OFFSET_HZ as f32,   // ← +75 kHz
));
// Pluto : modem-pluto/src/rx.rs (juste 0.0 pour le 3e param)
let mut chain = NbfmRxChain::new(NbfmRxChainConfig::new(
    session.negotiated_rate.sample_rate_hz as u32,
    session.rx_max_deviation_hz,
    0.0,                            // ← AD9363 fait la DC compensation
));

7. Tableau des paramètres par cas

Les paramètres dérivés (passband, stopband, nombre de taps, coût CPU) sont calculés automatiquement à partir des trois entrées (input_rate, max_deviation, lo_offset). Récap des cinq cas qui peuvent se présenter :

Cas Backend Mode input_rate max_dev lo_offset ÷N passband × 2 (BW canal) N taps CPU ~
1 SDRplay NBFM ±5 kHz 576 kHz 5 kHz +75 kHz 12 16 kHz 723 ~72 Mmac/s
2 SDRplay narrow ±2.5 kHz 576 kHz 2.5 kHz +75 kHz 12 11 kHz 1051 ~103 Mmac/s
3 Pluto NBFM ±5 kHz 576 kHz 5 kHz 0 12 16 kHz 723 ~70 Mmac/s
4 Pluto narrow ±2.5 kHz 576 kHz 2.5 kHz 0 12 11 kHz 1051 ~101 Mmac/s
5 Pluto fallback NBFM ±5 kHz 2304 kHz 5 kHz 0 48 16 kHz 2889 ~286 Mmac/s

À titre d'échelle : un cœur Pi 5 atteint 5 à 10 Gflops en f32, donc même le pire cas (Pluto fallback ≈ 286 Mmac/s) consomme moins de 5 % d'un cœur. Les machines x86 modernes ne le voient même pas.

8. Comment vérifier que ça marche

Tests automatiques en place

TestVérifie
kaiser_taps_meet_stopband_target Le filtre Kaiser atteint bien ≥ 75 dB de réjection au-delà de la stopband.
freq_xlating::adjacent_channel_rejected Un voisin à +25 kHz est atténué de ≥ 70 dB en sortie de FreqXlatingFir.
freq_xlating::lo_offset_brings_tone_to_dc Avec center_freq = +75 kHz, un tone à +75 kHz arrive à DC en sortie (vérifie le sens du NCO).
freq_xlating::image_at_minus_center_is_rejected Un tone à −75 kHz, image potentielle, est rejeté ≥ 70 dB (sens NCO + LPF).
nbfm_rx_chain::end_to_end_recovers_audio_with_lo_offset Cas SDRplay : NBFM PM-encodé à +75 kHz → audio recovered ±3 dB.
nbfm_rx_chain::end_to_end_recovers_audio_without_lo_offset Cas Pluto : même test avec offset = 0.
nbfm_rx_chain::adjacent_does_not_swamp_wanted Wanted + adjacent 20 dB plus fort à +25 kHz → SNR audio recovered > 20 dB.
nbfm_rx_chain::pluto_fallback_rate_works Cas 5 : input à 2304 kS/s, ÷48, tap count dans le budget.
nbfm_rx_chain::chunk_boundary_audio_matches Appel en deux moitiés ≡ appel en un bloc (état seamless).

Validation sur l'air

  1. SDRplay : taper la VFO sur la fréquence directe (par exemple 145.550 MHz). Avant le fix, il fallait décaler de ~50 kHz pour sortir du DC spike. Maintenant la réception doit être propre directement, et la sélectivité face à un voisin proche doit être nettement meilleure.
  2. Pluto : qualité au moins équivalente à avant. L'amélioration visible est la réjection des canaux adjacents en bande encombrée — un signal sur le canal voisin (25 kHz d'écart) doit être totalement absent du démod.
Diagnostic rapide en cas de réception cassée : si l'audio sonne comme avant (canal voisin présent, ou DC spike audible), vérifier dans cargo tree que modem-sdrplay et modem-pluto tirent bien la dernière version de modem-sdr-dsp (pas de cache de build périmé). Le point de vérité unique est le test adjacent_does_not_swamp_wanted — s'il passe, la chaîne elle-même est correcte.
