1. Pourquoi le retour TDM

La trame V4 initiale s'est calée sur le pattern pilotes sparse DVB-S2X (EN 302 307-1 §5.5.3) : un bloc de 36 symboles constants (1+j)/√2 après chaque LDPC codeword, densifié à 2 blocs/CW sur les profils APSK-32/64. C'était élégant — estimation LS de gain triviale (moyenne complexe), AGC neutre, distinct des points constellation — mais le bring-up OTA du 2026-05-13 sur la chaîne FTX-1 + SDRplay a montré un mode de panne franc :

  • V3 HighPlus (TDM 32 data + 2 pilotes / 22.7 ms anchor period) : σ² ≈ 0.011, décode propre.
  • V4 HighPlus2x (sparse blocs 2×36 / 178 ms anchor period) : σ² ≈ 0.10, ne converge pas.

Même chaîne hardware, même capture, facteur σ² × 10. La cause : le bruit de phase du transceiver + ADC soundcard occupe la bande 0.1–50 Hz ; le pattern TDM V3 le tracke (BW d'update ≈ 22 Hz, période pilote 22.7 ms), le pattern sparse-blocks ne le tracke pas (BW d'update ≈ 2.8 Hz, période pilote ~180 ms). La densité totale de pilotes était pourtant plus élevée sur V4 sparse (72 sym/CW pour HighPlus2x vs 30 en TDM), mais leur distribution temporelle laisse la phase walker entre deux ancrages.

Conclusion architecturale : la fréquence d'ancrage domine la densité totale pour suivre du bruit de phase basse fréquence. Le pattern V3 est de retour comme défaut, exposé per-profil comme PilotPattern2x { d_syms, p_syms } pour permettre de tuner densité et fréquence d'update séparément.

3. Trame V4 — cycle après refactor

La structure macro reste celle de la V4 (PLHEADER + LMS warmup + META-CW + DATA-CW × cw_per_cycle) ; seule la couche pilote au sein de chaque codeword change. Layout détaillé : 

cycle (≈ 4 s) =
    PLHEADER (192 sym)                             // SOF Chu 64 + PLS 128
      [LMS warmup (32 ou 64 sym, APSK-32/64 only)]
    META-CW interleavé TDM                          // 1 LDPC CW + pilots
    DATA-CW[0] interleavé TDM
    DATA-CW[1] interleavé TDM
    ...
    DATA-CW[k] interleavé TDM

La PLHEADER (SOF Frank-Zadoff-Chu 64 sym + PLS 128 sym Golay-encodé + redondance 12 sym profile_index) est inchangée. La rotation pilote est continue à travers le cycle : l'encodeur démarre la META-CW à group_idx_offset = 0, puis la k-ième DATA-CW à groups_per_cw·(1+k), sorte que le RX retrouve le pilote rotatif attendu en partant d'un seul compteur ancré sur la PLHEADER.

4. Pilotes TDM V3-style

Chaque LDPC codeword est interleavé en groupes d_syms data + p_syms pilotes, se répétant jusqu'à épuisement des data symboles du CW. Le dernier groupe peut être partiel côté data ; les pilotes sont toujours émis en entier (p_syms à chaque marque) : 

// Pattern par défaut (tous profils sauf HighPlusPlus2x)
PilotPattern2x::default_2x()   →  d_syms=32, p_syms=2

// Pattern densifié (HighPlusPlus2x, 64-APSK)
PilotPattern2x::dense_2x()     →  d_syms=16, p_syms=2

// Valeur pilote — rotation QPSK sur l'index absolu (continu cycle-wide)
pilot_symbol_2x(n) = exp(j·π/2·(n mod 4))
                   ∈ {(1,0), (0,1), (-1,0), (0,-1)}

4.1 Propriétés du pattern

  • Rotation QPSK identique à V3 — bit-for-bit compatible avec modem_core::pilot::pilot_symbol. Test pilot_bit_for_bit_v3_compatible.
  • Toujours sur le cercle unité — |pref| = 1 donc projection radiale/tangentielle triviale, pas de scaling.
  • Densité tunable par profil — PilotPattern2x est un champ de ModemConfig2x, pas une constante globale. Permet d'essayer 32/2, 16/2, 24/2, etc. sans recompiler la matrice de profils.
  • Période d'ancrage = (d+p)/Rs — à 1500 Bd avec 32/2 : 34/1500 ≈ 22.67 ms. BW d'update phase ≈ 1/(2·T) ≈ 22 Hz.
  • Estimation LS gain = Σ y · conj(pref) / Σ |pref|² sur toutes les positions pilote du CW + les références cycle-level (192 sym SOF+PLS partagés entre tous les CWs du cycle).
  • Estimation σ² split radial/tangentiel — résidu e_k = y_k/gain − pref_k, projeté sur conj(pref_k) : Re = radial (AM-AM), Im = tangentiel (phase noise / AM-PM). Sur AWGN pur les deux sont équilibrés (R/T ≈ 1) ; phase noise dominant donne R/T < 0.5.

6. σ² — estimation rotative

L'estimateur σ² ne change pas dans son architecture (split radial / tangentiel, médiane MAD, références cycle-level intégrées) ; ce qui change c'est la direction de projection, désormais portée par chaque pilote individuel (rotation QPSK) au lieu d'être constante conj((1+j)/√2). 

// rx_v4::estimate_cw_sigma2_split — schema
for chaque position pilote (p_start, p_end, abs_pilot_start) {
    for k in p_start..p_end {
        pref = pilot_symbol_2x(abs_pilot_start + (k - p_start))
        normed = chunk[k] / gain
        resid = normed - pref                  // dépend de la rotation
        proj  = resid * conj(pref)             // projection axe pref
        total_sq.push(|resid|²)
        radial_sq.push(proj.re²)               // axe pref
        tang_sq.push(proj.im²)                 // perpendiculaire pref
    }
}
+ cycle-refs (PLHEADER) — même schéma, direction = chaque sym SOF/PLS
σ²_{total,R,T} = median(.sq) / MAD_consistency, clampé à SIGMA2_FLOOR

Le ratio R/T = σ²_radial / σ²_tangential reste le diagnostic clé :

  • R/T ≈ 1 — canal AWGN isotrope (sim sans phase walk, OTA bonne qualité).
  • R/T < 0.5 — phase noise dominant (capture OTA FTX-1+SDRplay typique, sim avec phase walk).
  • R/T > 2 — AM-AM dominant (compression FM, rare en sortie de transceiver bien réglé).

8. Sim canal — modèle phase walk

L'IF AWGN seul ne reproduisait pas le mode de panne OTA : NBFM strippe le bruit de phase au limiter-discriminator. Le bruit phase qui dégrade la décode est post-démod, sur la porteuse audio du modem (centre 1500 Hz par défaut), composé de :

  • Jitter du clock ADC soundcard (~50 ppm RMS, mais surtout du jitter de phase instantané).
  • Phase noise LO close-in côté SDR (Lorentzian 1/f² typique, dominant en dessous de 100 Hz).
  • AM-PM résiduel du discriminator FM en régime non-linéaire (capture effect proche du seuil).

Modèle unifié : marche aléatoire brownienne sur l'enveloppe complexe audio. apply_audio_phase_noise() dans study/nbfm_channel_sim.py : 

# dphi_n ~ N(0, sigma² / sr)         (browian increment)
# phi(t) = cumsum(dphi)
# z = scipy.signal.hilbert(audio)     (analytic signal -> complex envelope)
# audio_out = Re(z * exp(j * phi))    (rotate envelope, take real part)

walk_std_per_sqrt_s est la std-dev de la marche en rad/√s — directement comparable aux specs Allan-deviation ou aux intégrales du phase noise L(f) du transceiver. Calibrations :

Sourcephw typiqueEffet sur HIGH+2X TDM
Aucun (sim AWGN pur)0.00 σ²=0.0015, R/T=1.10, décode propre
FTX-1 + soundcard nominal~0.05 σ²=0.0019, R/T=0.76, décode propre
FTX-1 + soundcard dégradé / SDRplay LO bon~0.10 σ²=0.0021, R/T=0.57, décode propre
SDRplay LO close-in dégradé~0.15-0.25 σ² 0.003–0.005, R/T < 0.3, marge LDPC sollicitée
SDR LO pire-cas (drift fort)> 0.40 décrochage, nécessite pattern dense 16/2 sur HIGH+

12. Conclusions architecturales

Au-delà des chiffres bruts, les leçons que ce refactor a produites — et qui devraient guider les choix de pilote sur les futurs profils (Pluto SDR, QO-100, autres canaux).

12.1 Bande passante d'ancrage > densité totale de pilotes

Le contre-exemple est HIGH2X : le pattern sparse blocks et le pattern TDM ont strictement le même nombre de symboles pilotes par CW (36). Seule la distribution temporelle change. Résultat : σ² ÷ 4 (+6 dB) en TDM. Plus parlant encore, HIGH+2X passe de 72 pilotes (sparse) à 30 (TDM) — moins de pilotes — et décode mieux dans 100 % des cas testés.

Théorème opérationnel : pour un budget pilote donné, l'optimum n'est pas de les concentrer en blocs (LS estimate plus précis localement) mais de les espacer aussi finement que la plus haute fréquence du bruit de phase à suivre. Pour NBFM amateur + SDR LO close-in, ce coin est à 20–50 Hz, ce qui impose un ancrage tous les 10–25 ms — soit notre 22 ms TDM.

12.2 R/T < 1 est le diagnostic en or

Le ratio σ²_radial / σ²_tangentiel, dispo gratuitement à chaque CW dès qu'on a 2 axes de pilotes projetés, donne le diagnostic mécanistique du canal en une seule mesure :

  • R/T ≈ 1 → AWGN isotrope, rien à voir, on est dans le modèle théorique.
  • R/T < 0.5 → bruit de phase non-tracké. Solution : densifier le pattern pilote (raccourcir la période d'ancrage).
  • R/T > 2 → AM-AM (compression FM, AGC). Solution : ajuster le hard-clip TX, vérifier la chaîne de gain RX.

Cette ligne est désormais sur la sortie CLI verbose, le bench snr_sweep_2x.py, et la mémoire visuelle du dev. Avant n'importe quel debug timing/LDPC, lire R/T donne la classe du problème.

12.3 Quand sim et OTA divergent par ×10, le terme manquant n'est pas l'implémentation

Le sim IF-AWGN seul donnait σ² ≈ 0.005 (HIGH+2X sparse). L'OTA donnait σ² ≈ 0.10. Facteur 20. La tentation : chercher un bug RX. La bonne réponse : le sim manque un term physique. NBFM strippe le bruit de phase IF au limiter-discriminator, donc la composante phase noise qui dégrade les ASPK n'arrivait jamais à la sortie du sim. Ajouter apply_audio_phase_noise() (15 LOC, brownian walk via Hilbert) a réconcilié sim et OTA.

Règle générale : quand un bench dit « tout va bien » mais que le banc réel hurle, chercher d'abord quel processus stochastique du canal réel a été supposé négligeable et ne l'est pas. Ici, c'était la densité spectrale de bruit de phase à offset < 100 Hz.

12.4 Configurer ce qu'on tune, pas ce qui découle

L'ancien pilot_blocks_per_cw: usize mélangeait deux axes physiquement distincts :

  • L'overhead (fraction de symboles pilote, lié à p_syms × n_groups / cw_wire).
  • La bande passante de tracking (fonction de (d_syms + p_syms) / Rs).

Le passage à PilotPattern2x { d_syms, p_syms } expose chaque axe séparément. On peut désormais :

  • Densifier la BW tracking sans toucher l'overhead — passer 32/2 à 16/2 double les ancrages et double l'overhead.
  • Réduire l'overhead sans toucher la BW — passer 32/2 à 32/1 garde les ancrages tous les 33 sym mais halve l'overhead (au prix d'une précision LS dégradée par √2).
  • Pousser HighPlusPlus2x dense à 16/2, garder HIGH+2X à 32/2 — chaque profil a son knob propre, géré dans profile2x.rs.

Pour le futur : un profil HIGH+PRO_2X pour Pluto SDR (canal LO plus bruité) pourrait choisir 24/2 (overhead intermédiaire, BW intermédiaire) — un seul champ à changer dans la table de profils.

12.5 Worktree A/B sur le même WAV = méthodologie de référence

Le facteur ÷5 σ² entre sparse-blocks et TDM à phw=0.10 a été démontré sur la même réalisation du canal (même seed, même bruit, même phase walk, même WAV TX). Méthodologie :

  1. git worktree add /tmp/control e90a8e4 pour l'ancien commit.
  2. Build CLI dans le worktree.
  3. Lancer le sweep depuis le checkout principal avec --nbfm-modem /tmp/control/rust/target/release/nbfm-modem, le canal sim étant fourni par le code HEAD.
  4. Comparer ligne à ligne du CSV.

Élimine d'un coup variance de seed, variance SNR, et toute dérive temporelle du sim setup. À adopter pour toute question « est-ce que ce changement aide vraiment ? ».

12.6 Le choix DVB-S2X n'était pas faux, juste pas-pour-ce-canal

DVB-S2X est optimisé pour des canaux satellite où la cohérence de phase est typiquement très longue (LO PLL spectralement propre, peu de phase noise close-in). Un bloc de 36 sym tous les 4 secondes est surdimensionné pour ces canaux. Sur NBFM amateur + SDR LO grand public, le profil de bruit de phase est tout autre : 1/f² close-in dominant, pas de PLL bench-grade. La spec DVB-S2X devient inadaptée.

Le pattern TDM V3 avait été dérivé empiriquement sur le banc amateur ; ce refactor + ce bench lui donnent une justification physique explicite. C'est l'asset le plus durable du travail : la grille densité × BW est désormais documentée et tunable, pas une supposition.

Une phrase. Le pattern pilote optimal n'est pas « celui qui ressemble à DVB-S2X » ni « celui qui maximise les pilotes par CW » — c'est celui dont la période est plus courte que l'inverse de la plus haute fréquence de bruit de phase à suivre, au-delà ça devient du gaspillage de débit.
