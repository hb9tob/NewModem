1. Vue d'ensemble — objectifs et choix architecturaux

Le modem 2x est une refonte du modem audio NBFM avec quatre objectifs explicites, chacun assorti d'une décision architecturale :

Objectif Décision Justification
Tracking drift horloge robuste (sound-card >50 ppm) Boucle fermée Farrow + Gardner remplace le resampling open-loop V3 Le V3 Phase 1d estime un drift global puis resample en bulk ; au-delà de 50 ppm l'erreur résiduelle accumule sur un cycle PLHEADER et fait perdre un CW sur quatre. Closed-loop suit en continu.
Late-entry rapide (joindre un burst en cours) PLHEADER 192 sym (SOF Chu + PLS Golay) toutes les ~4 s Chaque PLHEADER porte profile_index, base_esi, flags, frame_counter ; aucun état préalable nécessaire côté RX, le décodeur ressort prêt à consommer le cycle suivant.
Tracking bruit de phase soundcard (0.1–50 Hz) Pilotes TDM V3-style — 32 data + 2 pilotes / groupe (16+2 sur 64-APSK) Ancrage tous les 22 ms à 1500 Bd → BW d'update ~22 Hz, capable de suivre le bruit de phase FTX-1 + ADC. La variante DVB-S2X sparse blocks (36 sym tous les ~180 ms, BW ~3 Hz) avait échoué OTA — voir v4_tdm_architecture.html.
Décodage soft optimal sur APSK anisotrope LLR max-log per-ring Bayésien + turbo Pass 2 EM avec RTS smoother σ²(s) varie d'un facteur 17× entre anneau intérieur et extérieur sur HighPlus2x ; un LLR scalaire surestime la confiance des points proches de l'anneau extérieur. Le terme + log σ²(s) par candidat rétablit la marginale postérieure correcte.
Quoi reste de V3 ? LDPC WiMAX 802.16e N=2304 (rates 1/2, 3/4, 5/6) — inchangé. Constellations DVB-S2/S2X (QPSK, 8PSK, 16-APSK, 32-APSK, 64-APSK) — inchangé. RaptorQ fountain code (modem_framing::raptorq_codec) — inchangé. AppHeader meta CW (session_id, file_size, K, T, mime, hash) — inchangé, déplacé du marker control payload V3 au META-CW V4.

4. Trame V4 — spécification wire

4.1 PLHEADER — préambule double-Chu + PLS (256 sym, post-Phase 1)

256 symboles (était 192 avant Phase 1), émis au début de chaque cycle (~4 s nominal). Constants dans modem-core2x/src/plheader.rs:53-74. 

PLHEADER (256 sym total = 128 preamble + 128 PLS)
├─ PREAMBLE (128 sym = 2 × Chu_64) :
│       Deux séquences Frank-Zadoff-Chu IDENTIQUES, racine selon famille
│       {A=1, B=3, C=5}, choix selon sps = 48_000 / Rs :
│         sps ≥ 1200  → A   (Rs ≤ 40 Bd, jamais atteint en 2x)
│         750 ≤ sps < 1200 → B  (jamais atteint en 2x)
│         sps < 750   → C  (toutes les configs 2x actuelles)
│
│       Formule :  s[n] = exp(j · π · r · n² / 64),  n ∈ [0,63]
│                 preamble[n]    = s[n]
│                 preamble[64+n] = s[n]    // RÉPÉTITION exacte
│
│       Pourquoi 2 Chu :
│       (a) Détection RX par AUTO-CORRÉLATION Schmidl-Cox (= les deux
│           moitiés sont identiques → métrique |R|²/(P1·P2) ∈ [0,1] = 1
│           au préambule), invariante aux distortions AGC / LPF / CFO
│           qui affectent les deux moitiés identiquement.
│       (b) Cross-corrélation post-détection sur 128 sym au lieu de 64 :
│           gain de processing +3 dB (peak 128 vs 64).
│
└─ PLS  (128 sym)  :  payload 13 octets (11 info + 2 CRC16) → 88+16 = 104 bits
                      → padding à 108 bits → 9 blocs de 12 bits
                      → Golay(24,12) → 9 × 24 = 216 bits codés
                      → 108 symboles QPSK
                      + redondance 12 sym (re-encode bloc 0 contenant profile_index)
                      + zero-pad 8 sym
                      = 128 sym total

LS gain estimation au décode : moyenné sur les 128 sym de préambule
(au lieu de 64), σ²(gain) divisée par 2 vs l'ancienne version 192-sym.

  • SOF_LEN_SYM = 64 (longueur d'un seul Chu, inchangé)
  • PREAMBLE_LEN_SYM = 128 (= 2 × SOF_LEN_SYM, ajouté Phase 1)

4.3 Pilotes TDM — pattern et rotation

// pilot2x_tdm.rs
PilotPattern2x { d_syms: 32, p_syms: 2 }    // default — 7 profils sur 8
PilotPattern2x { d_syms: 16, p_syms: 2 }    // dense  — HighPlusPlus2x (64-APSK)

// Layout par LDPC CW (post-interleaver bits→sym) :
[d_syms data] [p_syms pilotes] [d_syms data] [p_syms pilotes] ...

// Valeur pilote rotative (cercle unité) :
pilot_symbol_2x(n) = exp(j · π/2 · (n mod 4))
                   ∈ { (1, 0), (0, 1), (-1, 0), (0, -1) }

Trois propriétés importantes :

  • Index pilote absolu. La rotation est continue à travers les groupes et les codewords d'un même cycle. L'encodeur démarre la META-CW avec group_idx_offset = 0, puis la k-ième DATA-CW avec groups_per_cw · (1 + k) frame2x.rs:374. Le RX reconstruit la séquence depuis un seul compteur ancré sur la PLHEADER.
  • Toujours unit-magnitude. Donc la projection radiale / tangentielle se calcule sans facteur de gain à compenser ; AbsGardner fonctionne directement (TED unbiased sur signal d'amplitude constante).
  • Période d'ancrage = (d+p)/Rs. À 1500 Bd avec 32/2 : 34/1500 ≈ 22.67 ms ⇒ BW d'update phase ≈ 22 Hz, suffisant pour suivre le bruit de phase basse-fréquence FTX-1 + ADC.

4.4 Pré-burst PRBS 2 s — AGC + FFE training (Phase 2)

preburst.rs. Émis par le TX au DÉBUT de chaque burst (avant la 1ère PLHEADER) quand vox_seconds > 0. Sert deux fonctions complémentaires :

  1. Stabilisation AGC du transceiver RX. Le FTX-1 (chaîne de référence, cf. feedback-reference-chain-991a-ftx1) a un slow-AGC à τ ≈ 1-3 s. Sans pré-burst, l'AGC compresse dynamiquement les premières dizaines de ms du PLHEADER, atténuant le pic Chu à ~20 % de sa valeur lab. 2 s de pré-burst (modulé identique à la data) laissent l'AGC converger à l'opérating-point cible avant que le PLHEADER1 arrive.
  2. One-shot LS-training du FFE. 3000 symboles connus (= 50× plus que les 224 sym des refs cycle-local PLHEADER+warmup) donnent un Gram matrix très over-determined pour FFE à 64 taps. Le FFE capture l'integral du group delay IF + multipath + non-flatness audio post-pre/de-emphasis, plus précis que le PLHEADER-only training.
// preburst.rs
PREBURST_LEN_SYM = 3000          // = 2.0 s à 1500 Bd, 3 s à 1000 Bd, 6 s à 500 Bd
LFSR_SEED        = 0x7FFF
LFSR polynôme    = x^15 + x + 1  // Galois right-shift, période 2^15-1 = 32767 bits

lfsr15_bits(6000)              // 6000 bits = 2 × PREBURST_LEN_SYM bits/sym (QPSK)
qpsk_gray().map_bits(bits)     // 3000 symboles QPSK Gray
reference_symbols()            // OnceLock-cached, partagé TX/RX

Au TX (modem2x.rs::encode_to_samples), le pré-burst s'insère entre VOX-tail silence et le data superframe : 

if vox_seconds > 0 {
    [VOX tone] + [VOX_TAIL_SILENCE 50 ms]
    + [PRBS preburst 3000 sym, modulé RRC + carrier identique à la data]
    + [data superframe]
    + [INTER_FRAME_SILENCE 200 ms] + [EOT] + [POST_EOT_SILENCE 100 ms]
}
// vox_seconds == 0 : data + silence + EOT seulement (tests synthétiques)

Au RX, le pré-burst est opportuniste : rx_v4.rs::try_gather_preburst_refs à chaque SOF détecté regarde les sof_at − PREBURST_LEN_SYM .. sof_at symboles du sym_buffer streamé. Si la cross-corrélation contre le PRBS de référence dépasse 0.5 (correlation coefficient bornée [0,1]), les 3000 sym connus sont ajoutés aux refs FFE LS pour ce cycle. Si pas détecté (late entry, ou vox_seconds=0), fallback per-cycle PLHEADER+warmup.

6.4 Détecteur Schmidl-Cox préambule (PreambleProbe2x, Phase 3)

Remplace la sonde FFT-template legacy par une auto-corrélation glissante sample-domain qui exploite la structure répétée du préambule double-Chu (4.1). gate2x.rs:60-230. La métrique est un coefficient de corrélation complexe au carré, bornée strictement ∈ [0, 1] par Cauchy-Schwarz : 

// Pipeline (par appel check())
1. Downmix à fc = DATA_CENTER_HZ = 1100 Hz :
   bb_raw[i] = audio[i] · exp(-j 2π fc i / fs)

2. LPF moving-average de largeur w = round(fs / (2·fc)) = 22 samples :
   bb[i] = (1/w) · Σ_{m=i-w+1..i} bb_raw[m]
   - Premier null à 2·fc = 2200 Hz (= bande ghost après downmix réel)
   - Suppresse l'image conjuguée du downmix de signal passband réel
   - Modem band ±900 Hz : passé avec ~−2.6 dB seulement

3. DC-block : soustraire mean(bb) de chaque sample
   - Annule les faux-positifs sur tons purs (VOX tone → DC après downmix)

4. Pour chaque (sps, anchor) bucket dans { (32,Normal2x), (48,Robust2x), (96,Ultra2x) } :
     L = SOF_LEN_SYM × sps      // longueur d'un Chu en audio samples
     R(k)  = Σ_{m=0..L-1}  bb*(k+m) · bb(k+m+L)         // auto-corrélation
     P1(k) = Σ_{m=0..L-1}  |bb(k+m)|²                    // power half 1
     P2(k) = Σ_{m=0..L-1}  |bb(k+m+L)|²                  // power half 2
     M(k)  = |R(k)|² / (P1(k) · P2(k))    ∈ [0, 1]      // Cauchy-Schwarz bound

5. Sliding update O(1) par sample :
     R(k+1)  = R(k)  + bb*(k+L)·bb(k+2L) − bb*(k)·bb(k+L)
     P1(k+1) = P1(k) + |bb(k+L)|²        − |bb(k)|²
     P2(k+1) = P2(k) + |bb(k+2L)|²       − |bb(k+L)|²

6. Anchor classification : pick bucket avec score = M · √L max
   (biais long-L pour résoudre l'ambiguïté Ultra preamble → Normal bucket
   = SC fire intra-symbol smoothness de l'Ultra preamble dans Normal-bucket L)

7. Threshold universel : PROBE_THRESHOLD_2X = 0.5
   - Noise baseline théorique (random complex Gaussian) : E[M] ≈ 1/L ≈ 5e-4 à L=2048
   - Signal propre : M → 1 au préambule (deux moitiés exactement identiques)
   - OTA réel : M ∈ [0.6, 0.95] selon SNR/canal
   - Marge ~30 dB sur le noise floor

Pourquoi Schmidl-Cox vs FFT cross-corr legacy :

Perturbation OTAFFT cross-corr (legacy)Schmidl-Cox
AGC gain riding (FTX-1 slow τ ≈ 1-3 s)Module l'amplitude pendant le SOF → peak réduit (15-20 vs 64 OTA)Les 2 moitiés sont multipliées par le MÊME gain → cancel dans |R|²/(P1·P2)

Coût : 1 downmix O(N) + 1 LPF moving-avg O(N) + 3 buckets × O(N) sliding sum. Total ≈ 5·N flops à N=19 200, soit ~1-2 ms sur Pi5 (vs ~25 ms pour le FFT legacy).

Connu — false-positives résiduels : la métrique peut crosser 0.5 sur des fenêtres de PRBS quand par hasard deux blocs de 64 sym sont quasi-orthogonaux à la moyenne (probabilité ~chi²-tail). Le bootstrap en aval filtre ça via la validation Golay+CRC du PLS, et la drop policy Phase 3 (validity-aware, voir §8) préserve les vrais PLHEADERs contre ce bruit.

Algorithme local-max (Phase 3) — find_all_sofs ne reporte plus la PREMIÈRE crossing du seuil dans chaque min_cycle_skip_syms window (= politique legacy, jusqu'à Phase 2). Elle reporte maintenant le LOCAL MAXIMUM par window : 

// Deux passes (la 1ère O(N·L), la 2nde O(N)) :
// Pass 1 — magnitude à chaque position
mags[k] = |Σ_{n=0..PREAMBLE_LEN_SYM-1}  symbols[k+n] · preamble[n].conj()|

// Pass 2 — walk en min_skip windows, emit local-max
skip = max(min_cycle_skip_syms, PREAMBLE_LEN_SYM)    // typiquement cycle/2
window_start = 0
while window_start <= end:
    window_end = min(window_start + skip, end+1)
    local_best_k, local_best_mag = argmax(mags[window_start..window_end])
    if local_best_mag >= threshold:
        out.push(local_best_k)
        window_start = local_best_k + skip
    else:
        window_start = window_end
Pourquoi local-max et pas first-crossing ? Sur le pré-burst PRBS, ~2 % des positions ont par hasard une cross-corrélation contre le préambule supérieure à 25.6 (= juste au seuil). Le first-crossing scheme legacy reportait ces false-positives en PREMIER dans leur window et sautait par-dessus les VRAIS PLHEADERs (correlation = 128) plus loin dans la même window. Cycle 0 systématiquement masqué. Le local-max scheme garantit que le vrai PLHEADER (corr 128) éclipse toujours les faux (corr ~25.6) dans la même window.

Pourquoi seuil 0.2 et pas 0.5 ? Sur WAV synthétique propre, l'autocorrélation du double-Chu peake à 128. Sur la chaîne OTA de référence (FT-991A → FTX-1 sound-card, cf. feedback-reference-chain-991a-ftx1), le pulse SOF est atténué de 70-80 % par AGC + LPF + de-emphasis : les pics réels OTA landent autour de 30-40 (+3 dB vs legacy single-Chu 15-20). Le seuil 0.2 × 128 = 25.6 admet ces pics tout en restant ~30 dB au-dessus du bruit rx_v4.rs:64-83.

Le décodeur PLHEADER decode_plheader_at() plheader.rs:365-409 :

  1. LS-fit du gain complexe sur les 128 sym de préambule (= deux Chu copies, gain estimate σ² divisé par 2 vs legacy 64-sym)
  2. Normalise les 128 sym PLS par ce gain (PLS slice à [PREAMBLE_LEN_SYM..PLHEADER_LEN_SYM])
  3. Démappe en bits via QPSK Gray-decode
  4. Décode les 9 blocs Golay(24,12)

7.5 LLR max-log per-ring (Bayésien APSK)

Pour AWGN anisotrope (σ²(s) dépend du candidat s), la log-vraisemblance correcte est : 

p(y | s) = 1 / (π · σ²(s)) · exp(-|y - s|² / σ²(s))
-log p(y | s) = |y - s|² / σ²(s)  +  log σ²(s)  +  const

// LLR max-log per-ring :  modem_core_base/soft_demod.rs:92-140
for k in 0..n_points {
    neg_log_p[k] = |y - constellation[k]|² / σ²_per_point[k]
                   + log_σ²_per_point[k]
}
for bit_pos in 0..bps {
    min_b0 = min over k where bit_map[k][bit_pos] == 0 of neg_log_p[k]
    min_b1 = min over k where bit_map[k][bit_pos] == 1 of neg_log_p[k]
    LLR[bit_pos] = (min_b1 - min_b0) as f32
}
Points clé : la division par σ²(s) se fait par candidat avant le min, et le terme + log σ²(s) est gardé. Bootstrap drop policy — validity-aware (Phase 3)

Quand aucune paire valide n'est trouvée (= peaks invalid, ou paire à mauvaise distance), la politique de drop post-Phase 3 est conservative :

  • Aucun peak du tout (probe false-fired sur bande bruit nue) → drop most of buffer, disarme le gate, retry depuis Idle.

Pourquoi cette politique : le pré-burst PRBS (Phase 2) avait exposé un bug FSM. Sur ~2 % des positions PRBS, la cross-corrélation contre le préambule dépasse le seuil par chance (correlation ≈ seuil 25.6 vs vrais PLHEADERs à 128). L'ancien drop policy (drop past peaks[1]+1) jetait cycle 0 quand peaks[1] tombait juste avant la vraie PLHEADER1. Le commit suivait sur cycle 1 (PLS.base_esi=8, seg_id=1), le RaptorQ remplissait packets 8+ → packets 0..7 manquants → bytes 0..1500 = zéros dans la sortie. Conservatif = pas de drop = pas de perte de cycle.

Mid-preburst entry (RX démarre quelque part dans le pré-burst) Fin de PRBS → PLHEADER1 → data... Bootstrap commit sur (PLHEADER1, PLHEADER2) comme early entry. Preburst FFE training partiel — si N < 3000 sym précédant PLHEADER1 sont dans sym_buffer, le LS uses ce qu'il y a (peut être suffisant si N > ~1000). Intermédiaire : AGC partiellement stabilisé, FFE training réduit. Cycle 0 décode quand même.
Late entry (RX démarre en plein milieu du burst, pré-burst manqué) data du cycle K → PLHEADER_{K+1} → data du cycle K+1 → ... Schmidl-Cox arme quand PLHEADER_{K+1} entre dans la tail. Bootstrap commit sur (PLHEADER_{K+1}, PLHEADER_{K+2}). Preburst FFE training échoue (corrélation PRBS contre data < 0.5) → fallback per-cycle FFE training (PLHEADER + LMS warmup, 224 sym, idem comportement legacy). Identique au modem pré-Phase-1-2-3 sur la robustesse, à la différence près que la 1ère PLHEADER décodée donne pls.base_esi > 0 et pls.seg_id > 0, donc le RaptorQ sait combien de packets manquent au début. Si suffisamment de packets sont reçus pendant la suite du burst, RaptorQ peut quand même reconstruire (overhead 30 % par défaut).

Mécanisme RX qui rend ça gratuit : aucun mode FSM dédié à "preamble-training". À chaque SOF détecté par find_all_sofs, la fonction try_gather_preburst_refs(symbols, sof_at) rx_v4.rs:1262-1310 regarde les sof_at − PREBURST_LEN_SYM .. sof_at symboles du sym_buffer streamé. Si la corrélation avec le PRBS de référence (LFSR-15) dépasse 0.5, les 3000 sym connus sont ajoutés aux refs train_ffe_ls pour ce cycle. Sinon, fallback per-cycle. Opportuniste, jamais bloquant.

Intégration de la boucle turbo dans le streaming

Le turbo Pass 2 EM (§7.6, RTS phase smoother + per-ring gain + log-σ²) s'insère dans la FSM au sein du même chunk-call après le décode LDPC Pass 1 du cycle courant. Cela garde la propriété streaming (O(audio_in) par chunk, état persistant entre chunks) : 

process_audio_chunk(samples):
    audio_buffer.extend(samples)
    if !bootstrap_committed:
        try_bootstrap_pair()                            // §8 Bootstrap
        return
    estimate_drift_from_raw()                           // raw PPM update
    streaming_dsp.feed(samples)                         // §3 Pipeline RX
    sym_buffer = streaming_dsp.sym_buffer().to_vec()
    equalize_symbols_per_cycle_from(sym_buffer, ...)    // §7.2 FFE Pass 1
    run_state_machine():
        Idle → try_acquire()    : find_next_sof + decode PLHEADER
        Acquiring → try_lock()  : commit & enter Locked
        Locked → drive_locked() : decode N CWs avec turbo Pass 2 EM par CW
            ├─ per_cycle_ffe_refresh    // gather_cycle_refs (incl. preburst si dispo)
            ├─ extract chunks → decode_one_cw (Pass 1 LDPC)
            ├─ turbo_refine_pass2(failed_cw_list)        // §7.6 RTS smoother + DD refs
            └─ emit CwConverged / SessionDecoded events
        Locked → Idle (à la fin du cycle, scan_cursor_abs += cycle_period_sym)

Pas de re-process batch : le sym_buffer est streamé incrémentalement, pas reconstruit à chaque chunk depuis l'audio_buffer growing (cf. feedback-full-streaming). Le turbo Pass 2 opère sur les chunks d'un cycle qui ont déjà été produits par le streaming pipeline — pas de re-MF, pas de re-decimate.

10. Analyse des points de dérive

Pour chaque paramètre tracké, on documente : quel mécanisme le suit, quelle plage il peut absorber, et où la boucle peut casser. C'est la liste des paramètres dont la dérive non-tracé peut faire chuter les performances du modem 2x.

10.1 Horloge symbole (ppm sample clock) et raw resampler

Tracking : LS slope-fit sur les positions SOF CRC-validées (estimate_drift_from_raw avec parabolic audio-rate refinement sur la double-Chu) → cached_drift_ppm forward-updated → ratio appliqué au raw resampler polyphase streaming en première étape du pipeline RX. Pas de Gardner closed-loop dans le chemin de production (streaming_dsp.rs n'a qu'un PolyphaseResampler + downmix + MF + decimate, pas de TED) — le streaming est piloté par cached_drift_ppm uniquement.

Raw resampler — architecture

PolyphaseResampler dans streaming_dsp.rs:55-128 : banc de filtres 32 taps × 1024 phases, sinc · Kaiser β = 8. Le banc est précalculé statiquement à la construction (build_polyphase_bank) :

  • 32 taps → ~80 dB stop-band, ~0.05 passband ripple, transition band ≈ 4·fs/N_TAPS = 6 kHz. Largement assez pour préserver la forme RRC sans tailler la bande modem.
  • 1024 sous-phases → résolution sub-sample ≈ 1/1024 sample (= 20 ns à 48 kHz). L'erreur de snapping de l'offset fractionnel requis vers la phase la plus proche est bien sous le noise floor du design FIR.
  • Coût steady-state : 32 multiplies-add par sample TX de sortie. À 48 kHz × 32 = 1.5 M flops/sec → ~1 % du budget Pi5.

Comment le raw resampler est piloté (réactivité à la dérive)

L'API est StreamingDsp::feed_audio(audio_buffer, audio_drained, drift_ppm). À chaque appel — c'est-à-dire à chaque chunk audio qui arrive — le caller (Rx2xSession::refresh_symbols) passe le cached_drift_ppm courant. Le resampler calcule ratio = 1 + drift_ppm × 1e-6 et continue à émettre des samples TX-time à partir de son curseur resampler_next_tx : 

// Pour chaque sample TX à émettre :
let j      = resampler_next_tx               // cumul output index
let rx_pos = j as f64 * ratio                // = position RX-time requise (cumul)
let rel    = rx_pos − audio_drained_samples  // → indice dans audio_buffer
let i_int  = rel.floor() as usize            // sample audio entier le plus proche
let phase  = round((rel − i_int) × N_PHASES) // sous-phase polyphase

resampled[j] = Σ_{n=0..32}  filter_bank[phase][n] · audio_buffer[i_int − 16 + n]

resampler_next_tx += 1

Quand cached_drift_ppm change entre deux chunks, le resampler n'est PAS reset ni rewindé. Il continue à partir de resampler_next_tx avec la NOUVELLE ratio. Conséquence : les samples déjà émis gardent la ratio qu'ils avaient à leur émission, les nouveaux samples utilisent la nouvelle ratio. Pas de sauts de phase sub-sample, pas de jitter discontinu. L'erreur résiduelle est intégrée par les estimators de phase aval (per-CW pilot LS + Pass 2 RTS smoother). Cf streaming_dsp.rs:260-280.

Updates de cached_drift_ppm :

Bootstrap commit
cached_drift_ppm = 0.0. La distance entre 2 SOFs (s2 − s1 vs cycle_sym) a une résolution de O(PREAMBLE_LEN_SYM/2) ≈ ±64 sym donnant une erreur ±2 % ppm si on l'utilisait directement ; on préfère démarrer à 0 et laisser la suite refiner. rx2x_session.rs:802-811
Steady-state, chaque chunk
estimate_drift_from_raw sur les SOF positions CRC-validées accumulées. Audio-rate parabolic refinement du peak Chu (commit 840ίac7 réutilisé pour double-Chu) → précision ~0.5 ppm sur ≥ 3 SOFs. Si l'estimate jump ≥ DRIFT_UPDATE_THRESHOLD_PPM = 0.5 ppm, le ratio change. Sinon, le ratio reste verrouillé. rx2x_session.rs:1408-1525
Plage acceptée
cached_drift_ppm.abs() ≤ 300 ppm. Au-dessus = LS overfit sur faux-positifs, rejet (caller garde l'ancienne valeur). rx_v4.rs:415-417
Forward only
Pas de RTS backward sur la trajectoire drift (stub T3 toujours non câblé post-Phase 3 — cf §11). Le dernier cycle d'un burst hérite de la valeur drift_ppm la plus récente, donc obsolète si l'horloge a dérivé pendant le burst. Pas un problème sur sound-card stable (drift < 30 ppm), problème potentiel sur clock SDR thermique avec drift ramps importants.

Initialisation en early entry vs late entry

L'initialisation du raw resampler est identique dans les deux cas. À StreamingDsp::new(cfg) :

  • resampler_next_tx = 0
  • last_drift_ppm = 0.0
  • resampled = [], resampled_start_abs = 0
  • Tous les buffers downstream (baseband, mf_output, sym_buffer) vides ; les delay lines (mf_state) à zéros.

La différence entre early et late entry est uniquement où tombe le 1er SOF dans le flux sym_buffer du resampler :

Cas1er audio sample vu par le resamplerPosition du 1er SOF dans sym_bufferConséquence drift
Early entry = 1er sample du burst TX (ou très proche) ≈ VOX·48000 + tail·48000 + PREBURST_LEN_SYM·sps + offset divisé par sps. Pour vox=0.5 : ~3849 sym. cached_drift_ppm démarre à 0, l'estimator a tout le burst pour converger. Drift bootstrap = 0 ppm (proche réalité si TX/RX horloges < 10 ppm). Steady-state LS le rafine.
Late entry = sample mid-burst arbitraire ≈ 6 sym (= début du buffer, où le SC a déclenché) cached_drift_ppm démarre à 0 mais le clock TX a déjà dérivé de Δ × T_late ppm·s au moment du late entry. L'estimator converge dès qu'il a ≥ 3 SOFs CRC-validés (= 3 cycles × 4 s = 12 s minimum). Pendant ce transient, Pass 2 RTS phase smoother absorbe le résiduel.

Pourquoi le resampler ne se soucie pas du late entry : il opère sur sa propre origine = le moment où la session RX a démarré. resampler_next_tx = 0 émet « TX-time index 0 » qui correspond à RX-time sample 0 = premier sample lu. Si le late entry survient T secondes après le début du burst TX, l'origin du resampler est décalée de T secondes par rapport à l'origine TX. Mais c'est transparent pour le downstream : le find_all_sofs / find_next_sof localise le 1er SOF où qu'il soit dans sym_buffer, et la FSM ancre tout le timing à partir de cette anchor SOF position.

Failure modes restants

Burst < 3 cycles (~12 s)
Pas assez de SOFs pour estimate_drift_from_raw → cached_drift_ppm reste à 0. Sur sound-card avec horloges typiques 10-30 ppm, ça passe. Sur OTA forte drift > 50 ppm, échec.
Drift > 300 ppm
L'estimator rejette (rx_v4.rs:415). cached_drift_ppm bloqué à la dernière valeur valide. Sur SDR clock thermique pas vu en pratique sur la chaîne FT-991A → FTX-1 référence.
Drift ramp rapide pendant le burst
L'estimator est forward-only ; le dernier cycle voit un drift obsolète. Mitigation Pass 2 RTS smoother sur la phase (absorbe jusqu'à ~1 ppm de drift résiduel par cycle). Limite : ramps > 2 ppm/cycle ne sont pas suivis. Stub T3 (RTS backward sur drift) résoudrait, non câblé post-Phase 3.
Bootstrap commit drift estimation
cached_drift_ppm = 0 au commit, pas la valeur dérivée de s2−s1. Le PREAMBLE_LEN_SYM/2 ≈ 64 sym de résolution sur s2−s1 donne ±64/5764 ≈ 1.1 % d'erreur, soit ±11 000 ppm d'incertitude — inutilisable. La parabolic-refined estimation post-commit converge sous 12 s.

10.2 Phase porteuse + fréquence résiduelle

Tracking : deux niveaux en cascade.

  1. Pass 1 — gain pilote LS scalaire par CW. Estime g = Σ y·conj(pref) / Σ |pref|² sur tous les pilotes du CW + références cycle-level. Phase moyenne du CW = arg(g).
  2. Pass 2 EM — RTS smoother 2-state {φ, ω}. Lissage Kalman forward + backward sur les soft symbols dérotés. Mesure θ_k = arg(y_k · conj(s̃_k)) avec R_k = σ²_tang / |s̃_k|².
Plage théorique
Aucune limite hard sur la phase (wrapping géré). En fréquence résiduelle, la PLS porte un coarse freq_offset 50 Hz step ; le reste est absorbé par l'état ω du Kalman (drift / symbole).
Failure modes
  • Pass 1 utilise un gain CW scalaire : sur des CWs longs (1224 sym Ultra, 491 sym HighPlus, ~800 ms), une dérive phase significative est moyenné et le résidu va dans σ²_tang. Cas pathologique : bruit de phase fort + CW long → σ² gonflé alors que le signal est récupérable avec un meilleur smoother.
  • RTS Pass 2 ne tourne que si Pass 1 a produit un posterior. Si Pass 1 sort un LLR aplati (decode hard-failure), le soft symbol E[s_k] est mal défini, et le RTS dérive sur le bruit pur.
  • q_φ = max(σ²_phi, 1e-8) — process noise tiré du modèle canal lui-même. Boucle : σ²_phi mal estimé (peu de pilotes) → q_φ mal choisi → smoother trop rigide ou trop souple → posterior biaisé.
  • Pas de portage phase entre cycles : la PLHEADER de chaque cycle réinitialise l'estimation. Une vraie continuité (Kalman "outer loop" cross-cycle) n'existe pas en 2x19 ; la dérive cumulée est resynchronisée toutes les 4 s, ce qui peut introduire un saut visible si le bruit de phase basse-fréquence a "marché" entre temps.
  • Wraparound : arg() wrap à ±π. Le RTS fait unwrap interne, mais sur burst boundary ou gap de mesures (P(z_k=r) < 1e-3 sur plusieurs symboles d'affilée), l'unwrap peut être déraillé. Aucune protection explicite en 2x19.

10.3 Estimation σ²

Tracking :

  1. Per-CW MAD sur pilotes (radial + tangential split) → modèle canal.
  2. Per-ring log-σ² random-walk smoother en Pass 2 EM.
Failure modes
  • Faible nombre de pilotes : HighPlus2x a ~30 pilotes/CW + 192 cycle-level. MAD sur 30 valeurs est bruité ; résultat sensible aux outliers (despite MAD = robust). Sur cycles courts (1 CW only sur Ultra), σ² very noisy.
  • Heavy-tailed assumption : MAD suppose Gaussien via consistency factor (·1/0.6745). Bruit de phase laisse des tails plus lourdes → σ² sous-estimé → LLR overconfident → mauvais LDPC.
  • Sanity blend [0.2, 5.0] : si σ²_DD / σ²_model déborde, fallback model. Mais le model lui-même peut être faux (split radial/tangential erroné) ; fallback peut masquer une meilleure estimation EM.
  • SIGMA2_FLOOR = 1e-3 hard-coded : sur signal très propre (lab WAV), σ² réel peut être < 1e-3, le plancher cap la confiance LLR. Pas de problème pratique (LDPC converge bien au-delà) mais empêche de mesurer le vrai SNR-floor du décodeur.

10.4 Gain per-ring

Tracking :

  1. Pass 1 DD-LS : estimate_per_ring_gain sur les symboles data hard-décidés.
  2. Pass 2 EM : random-walk Kalman per-ring × {real, imag}.
Failure modes
  • Misclassification d'anneau : sur ring boundary, hard-decision se trompe d'anneau, et la mesure pollue le LS de deux anneaux à la fois. Pass 2 EM utilise soft posteriors, donc moins fragile, mais la première itération dépend de Pass 1.
  • Posterior support gate 1e-3 : symboles peu informatifs ne contribuent pas, l'état propage. Si un anneau est rarement vu (extérieur sur QPSK-like APSK ?), son smoother conserve son état initial — peut diverger silencieusement.
  • Q = σ²_am + 1e-7 : process noise tiré de l'AM-AM excess. Si σ²_am est mal estimé (mauvais split radial), Q est mal choisi.
  • Pas de tracking inter-CW : chaque CW redémarre avec g_r(0) initialisé scalairement (gain CW pilote). Dérive gain inter-CW (saturation ampli qui chauffe sur burst long) non tracée.

10.5 FFE — équalisation canal

Tracking : LS forward train sur PLHEADER + LMS warmup, appliqué wholesale à un cycle. Pass 2 turbo-FFE (re-entrainement sur DD refs des CWs convergés) stub seulement.

Failure modes
  • Training set limité : ~224 refs sur APSK, ~192 sur QPSK/8PSK. Pour FFE_LEN_2X = 64, ratio 3.5×. Marginal — la matrice Gram peut être mal conditionnée sur certaines réalisations bruit/canal, et le LS bruite les taps.
  • Pilotes TDM exclus : positions dispersées brisent la cohérence LS. Les ~30 pilotes/CW × N_cw d'un cycle (potentiellement ~270 sur HighPlus) sont perdus comme training data.
  • Pas de tracking canal in-burst : taps figés sur un cycle (~4 s). Multipath fading rapide non capturé.
  • Pass 2 turbo-FFE stub : dd_refs_pool existe dans Rx2xSession mais n'est pas peuplé en slice 2x19. Opt-in via RX_V4_FFE_TURBO=1, version actuelle "simple all-converged-CW-syms" n'améliore pas sur le baseline.
  • Désactivé par défaut : RX_V4_NO_FFE est l'état par défaut (FFE skipped). On dépend entièrement de la réponse plate du canal NBFM audio.

10.6 Détection SOF

Tracking : cross-corrélation linéaire Chu reference contre sym_buffer, threshold 0.2 × SOF_LEN_SYM = 12.8.

Failure modes
  • OTA attenuation : NBFM demod + LPF + de-emphasis + AGC atténuent le pulse SOF de 70-80 %. Pics OTA réels landent 15-20 sur SDRplay. Threshold 12.8 admet ces pics mais la marge est faible : un trou de fading 3 dB de plus, on rate la SOF.
  • Faux positifs : threshold 12.8 sur 64 sym permet des pics bruit corrélé (pas blanc) à passer. PLS Golay + CRC16 rattrapent (taux faux positif négligeable côté PLS), mais on gâche du CPU sur des correlations qui échouent au décode.
  • Cross-family Chu sidelobes ≤ 8 : marge ~4× sous threshold. OK en théorie, OK en pratique mais à surveiller si on ajoute une famille D.
  • Drift > 1-2 % dans un cycle : le pic Chu se smear sur plusieurs sample shifts, magnitude divisée par √N_smear. À 50 ppm × 4 s × 1500 Bd = 0.3 sym de drift par cycle, négligeable. À 5000 ppm (très improbable, mais possible sur SDR mal référencé) : pic divisé par 2-3, on rate la SOF.
  • EOT over-read : sur le dernier cycle (FLAG2X_LAST), un sub-threshold tighter 0.7 × SOF_LEN_SYM = 45 détecte si on est en train de lire dans le silence post-burst rx_v4.rs:1478. Peut rater un cycle légitime si le SNR chute brutalement à la fin.

10.7 Continuité multi-burst

Tracking : état session persistant à travers les bursts (cw_bytes, drift_ppm, pls_anchors, dd_refs_pool).

Failure modes
  • Pas de gestion explicite de changement de session_id : la PLS porte session_id_low (8 bits faibles) mais le worker ne vérifie pas activement la cohérence entre bursts. Un changement de TX vers un autre opérateur avec ESIs qui se chevauchent corromprait cw_bytes.
  • Drift cached entre bursts : drift_ppm d'un burst précédent est appliqué au cubic resampling du nouveau burst, même si l'horloge TX a changé entre temps (nouveau radio amateur émet). Reset implicite seulement quand 3 nouvelles SOFs font un fit divergent.
  • Pas de timeout de session : un burst orphelin en mémoire indéfiniment. Pas critique côté correctness mais peut gonfler les buffers.
  • RaptorQ pool : cw_bytes persiste, donc un CW orphelin de burst n-1 peut être combiné avec ceux de burst n si les ESIs alignent par hasard. Garde : k_source et n_repair viennent de l'AppHeader du META-CW, qui change par session.

12.3 Lacunes identifiées post-Phase 3

Implémentation complète du plan ~/.claude/plans/pseudo-al-atoire-2s-poue-swirling-lemon.md, tous tests verts hors pré-existants. Les zones à creuser pour la suite :

(L1) Validation OTA pas encore faite

Le code Phase 1+2+3 n'a JAMAIS tourné OTA. Tout est validé sur sim audio synthétique. Premier OTA test prévu :

  • Build .deb 0.11.0-2x21 avec les changements Phase 1+2+3.
  • Deploy Pi5 kiosk, faire un TX OTA via FT-991A → FTX-1 sound-card (chaîne référence, cf. feedback-reference-chain-991a-ftx1).
  • Hypothèse : les pics Chu OTA doivent ~doubler (15-20 → 30-40, +3 dB processing gain du double-Chu) + l'AGC stabilizer pendant le 2 s de pré-burst doit éliminer la compression dynamique du SOF. Probe Schmidl-Cox metric attendu > 0.6 sur OTA propre.
  • Inspecter RX2X_LOG_PREBURST pour confirmer que le preburst-FFE training fire en early-entry.

(L2) Streaming Schmidl-Cox state pas persistant entre chunks

Le SC detector recompute les sommes R, P1, P2 from scratch à chaque appel sur les 400 ms de tail audio. Pas O(1)/sample cross-chunk. Pour HIGH+2X chunk=2400 samples, le SC re-process 19 200 samples × 3 buckets ≈ 60 K flops à chaque chunk, ~1-2 ms Pi5. Acceptable mais pas optimal. Future optimization : maintenir R, P1, P2 incrementally across chunks pour vrai O(1)/sample. Pas blocking. Cf. feedback-full-streaming.

(L3) `find_all_sofs` recompute correlations à chaque appel

L'algo local-max two-pass recompute toutes les magnitudes mags[k] à chaque try_bootstrap_pair. Sur bootstrap qui peut tourner plusieurs fois (= chunk-by-chunk en attendant valid pair), ça refait le même travail. Future optimization : cache mags[k] partiel et n'ajouter que les nouvelles positions. Pas blocking.

(L4) Turbo boucles T2/T3/T6/T7 toujours stubs

Section §11 inchangée par Phase 1-2-3. Voir aussi turbo-sync-next-session en mémoire pour le plan de fermeture turbo SYNC sample-domain.

(L5) RaptorQ recovery sur late entry (PLHEADER1 perdu)

En late entry, le burst commence à PLHEADER_{K} pour K > 0. Le RaptorQ base_esi > 0 → packets [0..K·cw_per_cycle) sont absents. Si le burst est assez long et l'overhead 30 % suffisant, RaptorQ reconstruct quand même. Sinon décode partiel. Future work : négocier le base_esi à 0 même late entry via wrapping ESI, ou bumper le repair overhead. Pas critique tant que OTA bursts sont assez longs.

(L6) Pas de test end-to-end via worker

Le test session_decodes_burst_with_preburst_high_2x drive Rx2xSession direct, pas via rx_worker2x::spawn. Cf. feedback-rx-tests-via-worker-chunks qui exige tous tests RX via worker en cpal-sized chunks. À faire : reformuler le test pour passer par le worker.

(L7) `try_gather_preburst_refs` correlation threshold magic 0.5

Le seuil 0.5 sur le coefficient de corrélation au carré |⟨y, p⟩|² / (‖y‖²·‖p‖²) est arbitraire. Pour OTA dégradé le préburst pourrait ne plus correler à 0.5. Future tuning post-OTA.
