1. Pourquoi le 2x — objectifs et héritage

Le modem 2x est une refonte parallèle du modem V3 dans le même workspace Rust, vivant dans les crates suffixées 2x (modem-core2x, modem-worker2x). Les modes legacy (V3) et 2x sont mutuellement exclusifs à l'exécution : le CLI les sélectionne via --family <legacy|2x> et la GUI les dispatche via is_2x_profile.

Les trois objectifs visés étaient :

  1. Overhead trame divisé par ~5. V3 dépense ~21 % du temps-symbole en non-payload (marqueurs, TDM pilots, runout, ré-insertion préambule). V4 vise ~4 % en s'inspirant de DVB-S2X EN 302 307-1 : un PLHEADER compact (192 sym) une fois par cycle de ~4 s, plus un bloc pilote sparse de 36 symboles après chaque LDPC codeword.
  2. Récupération d'horloge en boucle fermée. V3 utilisait un estimateur open-loop Phase 1d (mesurer le drift puis ré-échantillonner). V4 utilise la chaîne canonique Farrow + Gardner TED + PI loop, capable de suivre >100 ppm en continu sans accumulation, et qui prépare la migration QO-100 (CFO ±30–50 kHz + bruit de phase 10 GHz, voir qo100-cfo-readiness).
  3. Promotion 64-APSK hors expérimental. En V3, HIGH++ était gated derrière un flag ; en V4 les huit profils sont auto-détectés sans distinction.
Décisions cardinales validées dans ~/.claude/plans/je-voudrsis-edudier-la-precious-treasure.md :
  • Crates parallèles, pas de migration in-place de V3.
  • PLHEADER « inspired only » — on garde la chaîne Golay(24,12) du codebase plutôt que le PLSCODE biorthogonal (64,7) DVB-S2X.
  • HighPlusPlus2x (64-APSK) promu dans ALL_AUTO_DETECT.
  • Pilotes densifiés (2 blocs/CW) pour les APSK-32/64 où la distance min quadratique réduite exige un anchor plus serré.

Le streaming RX n'utilise plus de sliding-window (différence de fond avec V3) : Gardner produit chaque symbole une fois timing-correctement, donc rx_v4_symbols est appelé périodiquement sur un buffer qui ne fait que croître ; les re-décodes sont idempotents.

4. Trame V4 — PLHEADER, cycle, EOT

4.1 PLHEADER (192 sym = 64 SOF + 128 PLS)

ChampTailleContenuCodage
SOF (Start Of Frame) 64 sym Frank-Zadoff-Chu, root selon famille (A=1 sps≥1200, B=3 ROBUST, C=5 ULTRA) QPSK unit-circle, autocorr cyclique parfaite
PLS 128 sym Payload 13 B (104 bits + 4 zéros) → PlsPayload : profile_index, seg_id, session_id_low, base_esi (24 bits), flags, frame_counter, freq_offset 9 blocs Golay(24,12) → 216 bits → 108 sym QPSK ; + 12 sym redondance bloc 0 (profile_index) ; + 8 sym zéros

4.2 Cycle (≈ 4 s)

Un cycle PLHEADER est l'unité repérable côté RX. Layout : 

[PLHEADER 192 sym]
[LMS warmup 32 (APSK-32) ou 64 (APSK-64) sym, vide sinon]
[META-CW + pilot block(s)]                ← AppHeader×4 dans le META
[DATA-CW + pilot block(s)] × cw_per_cycle ← RaptorQ packets ESI base_esi..

Cible : V4_PREAMBLE_PERIOD_S = 4.0 s. Le nombre de DATA-CWs par cycle est calculé en remplissant le budget de 4 × Rs symboles après avoir consommé PLHEADER + warmup + META-CW : 

cw_per_cycle = max(1, (4·Rs − 192 − warmup − cw_with_pilots_meta) / cw_with_pilots)

4.3 Cycle EOT (End-Of-Transmission)

Une transmission se termine sur un cycle dédié avec :

  • PLHEADER avec flag FLAG2X_EOT (bit 1) levé.
  • Un META-CW portant une AppHeader sentinel (file_size = 0).
  • Aucun DATA-CW.
Bug subtil corrigé (897edb2). decode_one_cw écrasait l'AppHeader à chaque META-CW décodé ; l'EOT (toujours dernier) clobbrait donc la vraie AppHeader → reassembly RaptorQ avec n_source_cw=0 → buffer vide. Fix : conserver l'AppHeader existante quand la nouvelle a file_size=0. Test de régression : data_then_eot_preserves_real_app_header.

5. Blocs pilote sparse (1+j)/√2

Chaque LDPC codeword est suivi sur le fil de pilot_blocks_per_cw × 36 symboles identiques : 

P = (1 + j)/√2     |P| = 1     arg(P) = π/4

Propriétés exploitées par le RX :

  • AGC neutre — |P|² = 1 = Es, donc ces symboles n'ajoutent pas d'énergie supplémentaire.
  • Estimateur de gain trivial — moyenne complexe des 36 symboles ÷ P donne le complex gain de la CW.
  • Distinct des points data — sur 16/32/64-APSK, P est à |P|=1, entre les rings, hors de toute constellation data. Sur QPSK et 8PSK, c'est exactement un point data (le rayon à π/4).
  • Anchor TED pour D-1c-ii — l'intérieur strict d'un bloc (34 symboles sur 36) est garantie de magnitude constante, donc AbsGardner y est non-biaisé.

Pour les profils où la distance min quadratique est plus faible (APSK-32, APSK-64), la densité passe à 2 blocs/CW : le CW est splitté en deux moitiés (de tailles inégales si cw_data_syms est impair — Apsk32 : 461 = 230 + 231 → la seconde moitié absorbe le reste). Le RX retrouve les positions de manière déterministe via chunk_offset_is_pilot.

6. Récupération d'horloge fil-de-l'eau

Le streaming RX implémente la chaîne Farrow + Gardner TED + PI loop en boucle fermée. Quatre changements par rapport au chemin V3 batch :

  1. NCO à phase absolue — un compteur nco_base_sample: u64 est avancé chunk par chunk, donc la phase de la porteuse reste continue même quand le buffer audio interne est trimmé. Pas de phase-step à chaque chunk.
  2. RRC matched filter overlap-save — le buffer garde taps.len() + sps + marge samples de queue entre chunks pour absorber la zone warm-up de la FIR.
  3. Farrow cubic-Lagrange interp — chaque strobe échantillonne à une position fractionnaire strobe_pos sans ré-échantillonner l'audio entier. Cubic Lagrange est exact pour tout polynôme de degré ≤ 3 et reste à ≤ −50 dBc jusqu'à ~0.4·fs.
  4. PI loop sur erreur Gardner — gains DVB-S2 références Kp = 0.05, Ki = 0.005, soit une bande passante de boucle BL·Tsym ≈ 0.01 (lente, robuste au bruit). La correction ±correction est ajoutée au nominal sps entre chaque strobe.

6.1 Strobes de lead-in

Les ~6 premiers strobes (où la position EARLY de Farrow tombe dans la zone de warmup FIR) utilisent mf[k·sps] directement sans passer par Farrow, et ne touchent pas la TimingLoop. L'intégrateur démarre à 0 propre.

6.2 Mémoire bornée

Le buffer audio interne est trimmé après chaque chunk à taps.len() + sps + 4 samples ahead du prochain strobe. Pas de croissance non bornée même sur une capture longue (heures).

7. TED pilot-aided — phase D-1c-ii

Sur les constellations multi-ring APSK, AbsGardner (|y_late|² − |y_early|²) est biaisé par les données : la transition entre symboles de rings différents génère une erreur de timing fictive qui fait dériver l'intégrateur PI loop. Gardner pur (Re((y_late − y_early) · conj(y_on))) est moins précis sur PSK pur mais reste sans biais data.

La recette DVB-S2X §9.3.2 :

  1. Tracker en Gardner pur tant qu'on n'a pas verrouillé un SOF.
  2. Une fois le SOF localisé (via rx_v4.first_sof_at), ancrer le frontend avec align_to_sof(abs_sym_idx).
  3. Pré-calculer un cycle_pilot_map: Vec<bool> sur un cycle complet. map[(abs_idx − anchor) % cycle_len] dit si le strobe courant est sur un pilote.
  4. Sur l'intérieur strict d'un bloc pilote (les 34 symboles où abs_idx, abs_idx−1 et abs_idx+1 sont tous pilotes), la TED bascule sur AbsGardner. Le bord (1er et dernier symbole du bloc) reste sur Gardner — sinon les Farrow lookups EARLY/LATE à strobe ± sps/2 tomberaient sur la transition pilote/data et AbsGardner re-deviendrait biaisé par le saut d'amplitude.

Validation : test pilot_aided_at_100_ppm_apsk32_matches_gardner_only montre les deux modes décodent identiquement à 100 ppm sur HighPlus2x. Test pilot_aided_noise_free_apsk64 montre que l'activation du gate sur burst noise-free ne fait pas dériver l'intégrateur (|integ| < 1e-2). Sweep pilot_aided_all_apsk_profiles_decode_at_50_ppm sur les 5 profils APSK à +50 ppm : tous passent.

La promesse "4 % overhead" du plan initial s'évalue une fois retirée la part META-CW qui reste techniquement payload (juste pas application-payload) ; les chiffres bruts ci-dessus suréestiment donc l'overhead "structural" par rapport à DVB-S2X strict.

8.1 Décomposition du cycle par élément de trame

Pour chaque profil, où vont les symboles d'un cycle PLHEADER complet (PLHEADER + LMS warmup + META-CW + cw_per_cycle × DATA-CW + tous les blocs pilote) ? Cinq postes, en pourcentage du cycle total :

Profil PLHEADER(1) LMS warmup(2) META-CW(3) Pilotes DATA-CWs(4) LDPC parité(5) Info payload(6)
ULTRA2X 7.5 % 0 % 46.3 % 1.4 % 22.4 % 22.4 %
ROBUST2X 5.1 % 0 % 31.6 % 1.9 % 30.7 % 30.7 %
NORMAL2X 3.3 % 0 % 13.8 % 3.7 % 39.6 % 39.6 %
HIGH2X 3.4 % 0 % 10.7 % 5.1 % 20.2 % 60.6 %
HIGH+2X 3.5 % 0.6 % 9.6 % 11.7 % 18.7 % 56.0 %
HIGH++2X 3.4 % 1.1 % 8.0 % 13.8 % 18.4 % 55.3 %
HIGH56_2X 3.4 % 0 % 10.7 % 5.1 % 13.5 % 67.4 %
HIGH+56_2X 3.5 % 0.6 % 9.6 % 11.7 % 12.5 % 62.3 %

Légende et formules (vérifiées : somme = 100 % à chaque ligne) :

  1. PLHEADER — 192 sym fixes (64 SOF Chu + 128 PLS Golay×9+CRC16). Coût d'amorçage par cycle ; s'amortit quand cw_per_cycle grandit (ULTRA2X paie 7.5 % parce que Rs=500 force cw_per_cycle=1).
  2. LMS warmup — 0 / 32 / 64 sym selon la constellation (QPSK/8PSK/16-APSK : 0 ; 32-APSK : 32 ; 64-APSK : 64). Seed le FFE audio sur des rings que la SOF QPSK ne touche pas.
  3. META-CW — un LDPC codeword complet (cw_data_syms) + son ou ses blocs pilote, dédié à porter AppHeader×4 (filename, callsign, MIME, file_size, ...). Coût constant par cycle ; pèse lourd sur ULTRA/ROBUST où le cycle est court (1-2 DATA-CWs) et léger sur les profils HIGH où on rentabilise sur 8-11 DATA-CWs.
  4. Pilotes DATA-CWs — somme des blocs pilote attachés aux DATA-CWs : cw_per_cycle × pilot_blocks_per_cw × 36. Doublé pour HIGH+/HIGH++/HIGH+56 (2 blocs/CW).
  5. LDPC parité — fraction (1 − rate) des symboles DATA-CW (les bits de parité du code LDPC). 50 % pour rate 1/2, 25 % pour rate 3/4, 16.7 % pour rate 5/6.
  6. Info payload — fraction rate des symboles DATA-CW = les bits d'information transportés. C'est l'information utile avant RaptorQ ; après codage RaptorQ (avec sur-dimensionnement typique ~5 %), le payload application réel est ~95 % de cette colonne.

Lectures clés :

  • Sur ULTRA2X, la META-CW à elle seule mange 46 % du cycle — conséquence directe du choix «cw_per_cycle ≥ 1 toujours» pour préserver la robustesse late-entry sur signal très faible.
  • Le passage de HIGH2X à HIGH56_2X (même 16-APSK, rate 3/4 → 5/6) gagne +7 points sur l'info payload (60.6 → 67.4 %) en sacrifiant 7 points sur la parité (20.2 → 13.5 %). Bonne piste si le canal est propre.
  • Les pilotes doublés sur les APSK haute-densité (HIGH+/HIGH++/HIGH+56) ajoutent +6.5 points d'overhead par rapport au choix 1 bloc/CW, mais sont indispensables pour la traque de phase sur les rings serrés (voir §5 et §7).
  • L'info payload maximale est obtenue avec HIGH56_2X à 67.4 % — pas avec HIGH++2X (55.3 %) malgré ses 6 bits/sym, parce que la densification pilotes +LMS warmup mangent une grosse partie du gain.

9. σ² — estimation, budget, lecture

9.1 Définition

Le σ² rapporté dans RxResult2x.sigma2_data est la variance résiduelle des symboles pilote après division par le gain estimé, moyenné sur les CWs d'un burst : 

// rx_v4::estimate_cw_sigma2 (par CW)
for chaque symbole pilote p_recu dans la CW {
    p_normé = p_recu / gain_estimé_CW
    sum_sq += |p_normé − P|²       // P = (1+j)/√2
}
σ²_CW = sum_sq / N_pilotes

// agrégation
σ²_data = mean(σ²_CW pour chaque CW data) clampé à SIGMA2_FLOOR=1e-3

L'unité est variance par dimension complexe pour Es = 1, donc directement comparable à un Es/N0 : 

Es/N0 [dB] ≈ −10·log10(σ²)

9.2 Utilisation : échelle des LLR

Le σ² par CW est passé à soft_demod::llr_maxlog : 

LLR_k(y) = (min_{s:b_k=1} |y−s|² − min_{s:b_k=0} |y−s|²) / σ²

Sous-estimer σ² produit des LLR sur-confiants → le LDPC fait des décisions précoces sur du bruit. Sur-estimer σ² aplatit les LLR → le décodeur converge plus lentement ou rate des erreurs corrigibles. La chaîne est donc directement sensible à la qualité de l'estimation pilote, ce qui motive la densification 2 blocs/CW sur les APSK-32/64.

10.3 Sweep canal NBFM — HIGH / HIGH+ / HIGH++ à 4 SNRs, payload 10 kB

Pourquoi V4 a un plancher σ² plus haut ? Deux causes plausibles :

  • V3 utilise des pilotes TDM distribués à l'intérieur de chaque codeword (toutes les 32-64 symboles selon le profil) — moyennés sur ~30 % du CW, ils donnent un estimateur de gain LS « chauffant » sur la totalité de la durée du CW. V4 a un bloc pilote sparse de 36 symboles après chaque CW : 36 échantillons est moins de samples pour l'estimateur que ~150-200 chez V3, et ils sont concentrés à un instant, pas répartis (plus sensible à un fading rapide non-stationnaire dans la fenêtre).
  • V4 utilise une chaîne timing closed-loop (Farrow + Gardner + PI) qui ajoute une jitter résiduelle de timing aux strobes ; sur 64-APSK où l'écart inter-rings est serré, cette jitter se traduit en variance constellation non-prise-en-compte par l'estimateur de bruit pilote.

Implications pratiques :

  • Sur ce canal NBFM simulé (sans drift d'horloge, AWGN seul), V3 HIGH++ a une marge SNR ~5-6 dB supérieure à V4 HIGH++2X pour la même charge utile. Si on veut le profil 64- APSK maximally robust, V3 reste le plus fiable.
  • V4 ajoute en contrepartie : la tracking d'horloge continue (qui protège des drifts >50 ppm que V3 ne supporte pas), un overhead trame ÷2 vs V3 (voir §2 et §8), et une PLHEADER courte de 192 sym qui se retrouve plus vite après une perte de synchro. Le compromis vise sound-card avec drift et QO-100 SDR avec CFO, pas la performance σ² pure sur canal stable.
  • Le gap σ² peut probablement être fermé en : (a) densifiant les pilotes V4 (passer de 36 à 64-72 syms par bloc sur HIGH++/HIGH+) ; (b) LS par ring séparé au lieu d'un LS global, qui éliminerait le biais data→pilote suspecté en §10.3 ; (c) activer pilot-aided dans la TimingLoop (déjà fait en D-1c-ii) pour réduire la jitter de timing sur APSK haute densité. Investigations TODO post-merge feat/modem-2x.

10.3.2 Décomposition radiale/tangentielle des pilotes — extraction des stats canal

Chaque résidu pilote e_k = y_k / gain − P est un nombre complexe. On le projette sur la direction de la pilote P = (1+j)/√2 : 

e_radial     = Re(e · conj(P))     ← composante le long de P
e_tangential = Im(e · conj(P))     ← composante perpendiculaire à P

Puis estimateur MAD par axe avec la constante de cohérence χ²(1) à 50% (≈ 0.4549, soit Φ⁻¹(0.75)² pour la loi normale standard, valide pour des résidus réels gaussiens 1D). L'output : σ²_radial et σ²_tangential indépendants. Sur AWGN pur les deux sont égaux ; tout écart révèle une structure non-AWGN du canal.

Lecture du canal NBFM sur 32-APSK — le ratio R/T de HIGH+2X est 53× à if_noise=0 et reste > 30× jusqu'à if_noise=0.45. La composante tangentielle est à σ²_T ≈ 6·10⁻⁴ (~32 dB SNR) — un canal quasi-parfait en phase. L'erreur résiduelle est presque entièrement radiale : c'est de l'AM-AM, conséquence de la fuite RRC des data symboles à magnitude ≈ pilote (R₂/γ₂ pour 32-APSK) qui contamine l'amplitude perçue des pilotes. La distorsion est structurée (corrélée au pattern data voisin), pas du bruit additif blanc.

Signatures par profil :

  • HIGH2X (16-APSK) : R/T ≈ 0.39 — tangentielle dominante. La pilote à |P|=1 est au-dessus des 2 rings 16-APSK (R₀=0.45, R₁=1.27·R₀=0.57·1=... attendre non : R₀=0.45·1=0.45, R₁=γ·R₀=2.85·0.198=0.564 — la pilote |P|=1 est hors du domaine data), donc peu d'ISI radiale. Le résidu vient surtout de phase noise FM / drift résiduel — d'où la dominance tangentielle.
  • HIGH+2X (32-APSK) : R/T > 30× — radiale écrasante. Pilote |P|=1 sandwich entre R₂=0.69 et R₃=1.27. Fuite RRC depuis les data symboles à |s|=R_outer=1.27 tombe en phase avec la pilote (mêmes angles π/4 sur le ring extérieur), donc bias purement radial.
  • HIGH++2X (64-APSK) : R/T ≈ 3 — radiale dominante mais pas extrême. Plus de rings, donc plus d'angles relatifs entre data voisin et pilote → mélange d'erreurs radiale et tangentielle.

Implications pour les fixes prévues :

  • La fix #2 §10.5 (SSD per-ring LS gain) attaque l'AM-AM par anneau, donc devrait surtout aider la composante radiale. Sur le canal NBFM simulé son effet est marginal (la distorsion AM-AM est faible). Sur un canal OTA réel où le transceiver compresse plus, le gain devrait être plus visible.
  • La fix #3 §10.5 (densifier pilotes) attaque le bruit d'estimation pilote, indépendant de R vs T. Donnerait un gain proportionnel sur les deux.
  • Une fix future spécifique 32-APSK serait filtrage de pilote temporel : convoluer les pilotes du bloc avec un noyau passe-bas pour atténuer l'ISI radiale stricture, qui est haute-fréquence par symbole. Atypique mais ciblerait précisément l'origine du R/T=53.

10.3.3 Décodeur turbo — deux passes LDPC avec affinage du modèle canal

Le décodeur RX-V4 est désormais à deux passes, selon le schéma demandé : 

Pass 1 (model-driven)
   ├── LS gain pilote → g
   ├── MAD radial/tangentiel sur pilotes (après rotation par g̅)
   │     → σ²_radial_pilot, σ²_tangential_pilot
   ├── ChannelModel { σ²_awgn_per_dim, σ²_am, σ²_phi }
   │     σ²_awgn_per_dim = min(σ²_radial_pilot, σ²_tangential_pilot)
   │     σ²_am   = σ²_radial_pilot − σ²_awgn_per_dim   (excès radial)
   │     σ²_phi  = σ²_tangential_pilot − σ²_awgn_per_dim
   ├── σ²_r par anneau via modèle :
   │     σ²_total(R) = 2·σ²_awgn_per_dim + R²·(σ²_am + σ²_phi)
   ├── llr_maxlog_per_ring(data, constellation, σ²_per_ring)
   │     σ²_per_symbol[i] = σ²_r(ring(s_hat[i]))
   └── LDPC → (info_bytes_p1, converged_p1)

Pass 2 (data-driven, gated on Pass 1 convergence)
   ├── Re-encode info_bytes_p1 → ŝ_k via
   │     frame2x::encode_one_codeword (vérité absolue grâce
   │     au LDPC syndrome check ≈ sans erreur sur N=2304)
   ├── Hard-assign anneau pour chaque ŝ_k (nearest constellation point)
   ├── MAD per-ring sur les résidus e_i = data_norm[i] − ŝ_k
   │     σ²_r,DD = median(|e|²) / ln(2)   (rings avec ≥ 8 syms)
   ├── Sanity check : si σ²_r,DD / σ²_r,model ∉ [0.2, 5.0]
   │     → fallback model σ² pour cet anneau (re-encode pathologique)
   ├── llr_maxlog_per_ring(data, constellation, σ²_per_ring_blended)
   └── LDPC → (info_bytes_p2, converged_p2)
       Si converged_p2 : trust pass 2 (LLR plus fidèle au canal réel)
       Sinon : fallback pass 1 (qui était déjà OK)
Note importante sur la formule LLR. Quand σ² dépend du candidat (chaque anneau a sa variance), la division doit être faite par candidat avant le min, pas après : 
log p(y|s) = −|y−s|²/σ²(s) − log σ²(s) + const

LLR(b_k) = min{s:bk=1} [|y−s|²/σ²(s) + log σ²(s)]
        −  min{s:bk=0} [|y−s|²/σ²(s) + log σ²(s)]
Le terme log σ²(s) est le prior Bayésien sur l'hypothèse de bruit : un candidat à haute σ² a une PDF de bruit plus large, donc moins probable à distance égale. Souvent négligé en "max-log lite", mais important quand σ² varie de plus de ~×2 entre anneaux — typique sur APSK multi-rings (HIGH+2X observation : σ²_outer/σ²_inner ≈ 17× après channel model). La fonction llr_maxlog_per_ring dans modem-core-base fait la division per-candidat avant min et inclut le terme log σ²(s). Le test llr_maxlog_per_ring_matches_scalar_when_sigma2_uniform vérifie la consistance avec llr_maxlog scalaire quand σ² est uniforme.
Pourquoi la même décodeur ré-itère sur la même donnée ? Sur les CWs déjà convergés en pass 1, pass 2 reconverge presque toujours aux mêmes bytes — la valeur ajoutée est dans la fidélité des LLR par anneau. Sur un canal où la σ²_r réelle diverge fortement entre anneaux (HIGH+2X avec R/T=53 sur l'anneau pilote → AM-AM massif sur les autres anneaux), le Pass 1 model-based donne déjà une bonne première approximation ; Pass 2 raffine sur la base de la vraie sortie LDPC. Bénéfice secondaire : reporting plus précis du σ²_r par anneau (utile pour le link quality monitoring) et un filet de sécurité contre les cas marginaux où pass 1 converge sur un mauvais codeword (rare pour N=2304, mais le pass 2 le détecterait).

Impact mesuré sur le sweep §10.3 (NBFM simulé, seed=100, 10 kB) — avec Pass 2 vs Pass 1 model-only :

  • HIGH++2X @ if_noise=0.00 : 63/69 → 64/69 CWs convergés (+1, modeste — le canal NBFM simulé a peu d'AM-AM, donc le modèle pilote-only est déjà ~bon).
  • HIGH+2X / HIGH2X : convergence inchangée (déjà max, 68/69 et 69/70 respectivement à ces SNRs).
  • Test unitaire dédié : turbo_pass2_handles_strong_per_ring_distortion_apsk32 applique 3 gains ring-dépendants (inner 0.90+0.05j, middle 1.05−0.03j, outer 1.15−0.10j) et vérifie le byte-exact — passe avec turbo, démontre que la chaîne complète Pass 1 + SSD + Pass 2 absorbe la distorsion APSK structurée.
  • Coût compute : ~1 LDPC iter additionnelle par CW convergé. Wall-clock par CW passe de 2.5s à 2.9-3.1s sur Pi5 (en debug), négligeable en release.

Le bénéfice plein du turbo se manifestera sur des canaux réels (transceiver OTA avec AM-AM/AM-PM non triviale) où le modèle pilote- only sous-estime la variance des anneaux extérieurs. La sanity check [0.2, 5.0] protège contre les divergences pathologiques.

12.1 CLI smoke roundtrip

cd ~/git/NewModem/rust
cargo build -p modem-cli --release

BIN=./target/release/nbfm-modem
TMP=$(mktemp -d)
head -c 600 /dev/urandom > "$TMP/in.bin"
for p in ULTRA2X ROBUST2X NORMAL2X HIGH2X HIGH+2X HIGH++2X HIGH56_2X HIGH+56_2X; do
  $BIN tx -i "$TMP/in.bin" -o "$TMP/$p.wav" --family 2x -p "$p" \
       --callsign HB9TOB --vox 0 >/dev/null
  $BIN rx -i "$TMP/$p.wav" -o "$TMP/$p.out" --family 2x -p "$p" >/dev/null
  cmp -s "$TMP/in.bin" "$TMP/$p.out" && echo "OK $p"
done
